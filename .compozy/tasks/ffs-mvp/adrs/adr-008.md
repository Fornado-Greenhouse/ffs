# ADR-008: Speak MCP and A2A at boundaries

## Status

Accepted

## Date

2026-05-05

## Context

As of April 2026, two protocol standards have converged with substantial industry adoption under the Linux Foundation:

- **MCP (Model Context Protocol).** Originated by Anthropic in November 2024; standardizes how AI agents talk to tools, data sources, and APIs. The "USB-C port for AI." Donated to the Linux Foundation; adopted by every major model provider.
- **A2A (Agent2Agent).** Originated by Google in April 2025; standardizes how agents communicate and coordinate with each other across organizational boundaries. Donated to the Linux Foundation in June 2025; absorbed IBM's competing ACP standard in August 2025. As of April 2026: 150+ organizations, v1.2 with cryptographically signed Agent Cards, production at Microsoft, AWS, Salesforce, SAP, ServiceNow.

FFS has a choice: integrate with these standards as boundary protocols, or invent its own.

The lesson from ACP-the-protocol's fate is instructive: standalone agent communication standards lose to the one with industry consensus. Anyone evaluating FFS will reasonably ask "do you work with MCP and A2A?"

## Decision

FFS speaks the standards at boundaries; FFS-native semantics live everywhere else inside the substrate.

- **MCP at the agent-to-tool boundary.** Any AI agent reads or writes the substrate via the FFS-MCP server (Phase 2 — see ADR-010).
- **A2A at the agent-to-agent boundary.** When FFS couriers federate to non-FFS peers (Phase 2+), they speak A2A wire format with FFS-shaped payloads.
- **FFS-native between FFS substrates.** Federation between two FFS graphs uses FFS bridge contracts directly; no need to round-trip through A2A when both sides understand the substrate semantics.

A2A's signed Agent Card pattern aligns conceptually with FFS's self-certifying graph IDs and key histories. Where reasonable, FFS author identities are made A2A-recognizable so an FFS-authored claim can be verified by an A2A peer without FFS-specific knowledge.

## Alternatives Considered

### Alternative 1: Invent FFS's own protocols

- **Description:** Define an FFS-specific protocol for agent communication and tool access. Don't depend on MCP or A2A.
- **Pros:** Full design control. No external standards to track.
- **Cons:** Repeats ACP's mistake. Standalone protocols lose to the ones with 150 organizations behind them. Cuts off the ecosystem of MCP-aware agents and A2A-aware agents that would otherwise be FFS-aware automatically.
- **Why rejected:** Strategic mistake confirmed by recent history. The leverage from speaking the standards is enormous; the cost of speaking them is small.

### Alternative 2: Speak MCP and A2A everywhere, abandon FFS-native

- **Description:** Replace FFS bridge contracts with A2A; replace local API with MCP. Use the standards everywhere.
- **Pros:** Maximum interop. No FFS-specific machinery to learn.
- **Cons:** MCP and A2A don't address what FFS is for — bitemporal claims, sub-record classification, capability-as-data, federation contracts, predicate grammar. Forcing these into MCP tools or A2A messages would lose semantics. The standards are at boundaries; FFS-native is for the inside.
- **Why rejected:** The standards don't provide what FFS does. They compose; they don't substitute.

## Consequences

### Positive

- Any MCP-aware agent (Claude, ChatGPT, Gemini, framework-agnostic agents) can read and write the substrate without bespoke integration once Phase 2 ships the MCP server.
- A2A-speaking peers can federate to FFS substrates when Phase 2+ adds the A2A bridge.
- FFS doesn't compete with MCP or A2A at the protocol layer; it composes. This avoids unwinnable fights against industry-standard protocols.
- The "speak the standards at boundaries" framing is legible to anyone evaluating FFS who knows the protocol landscape.

### Negative

- FFS depends on MCP and A2A continuing to be the converged standards. If they fragment or are replaced, FFS's integration story shifts.
- The MCP server and A2A bridge are real Phase 2 work. They're not free.

### Risks

- MCP or A2A evolves in ways incompatible with FFS's semantics. **Mitigation:** Both protocols are now under Linux Foundation governance with public roadmaps. Major changes are negotiated in the open.
- A new standard emerges and supersedes them. **Mitigation:** FFS's protocol integration is at boundaries, not in the substrate. Replacing the boundary protocol is a contained change.

## Implementation Notes

The FFS-MCP server is Phase 2 (see ADR-010). It exposes substrate query, projection rendering, atom authoring (capability-checked), `ffs://` URL resolution, predicate inspection, and audit query as MCP tools.

The A2A bridge is Phase 2+ work, contingent on a non-FFS counterparty wanting to federate. The bridge contract grammar maps FFS classification, capability, and erasure semantics onto A2A Agent Card capabilities lossily; what is preserved versus dropped is an open question (see PRD § Open Questions).

Inside the substrate, FFS-native semantics are uniform. The architecture does not invent protocols where the industry has converged on standards; it adds substrate semantics that those protocols don't address.
