# ADR-016: Single SQLite database per substrate with normalized atom store

## Status

Accepted

## Date

2026-05-05

## Context

PRD non-goals state SQLite is the MVP storage and alternatives (RuVector / RVF) are Phase 3+. The remaining question is the schema shape: how atoms, capabilities, classification claims, supersession chains, and provenance are organized so that bitemporal queries (`?as_of=...`, `?valid_at=...`) are fast, capability evaluation is correct, and the projection-rendering critical path stays under the PRD's 500ms budget.

Atoms have multi-axis access patterns:

- By content hash (atom addressing in `ffs://`).
- By entity (entity addressing, projection rendering).
- By predicate (path queries: contacts, people, notes).
- By bitemporal range (`as_of` historical queries).
- By supersession chain (resolving "current" claim from a chain).
- By classification (capability filtering, tier-based sharing).
- By author (audit, federation provenance).

Per-cohort encryption (DEK in OS keychain) was decided at the PRD level. Whether encryption is per-database (SQLCipher) or per-payload (application-layer AEAD on the atom blob) is open.

## Decision

The MVP atom store is a single SQLite database per FFS substrate, located at `~/.ffs/store.db`, encrypted at rest via SQLCipher using a DEK held in the OS keychain. The schema is normalized:

- `atoms` — content_hash (PK, BLOB), entity_id (BLOB), predicate (TEXT), author (BLOB), valid_from (INTEGER), valid_to (INTEGER, NULL = open), tx_time (INTEGER), classification_id (FK), supersedes (BLOB, NULL = root), signature (BLOB), envelope (BLOB, the canonical-JSON atom bytes). Indexes on (entity_id, predicate, tx_time DESC), (predicate, tx_time DESC), (supersedes), (author, tx_time DESC).
- `classifications` — id (PK), atom_hash (FK), tier (TEXT), classifier (BLOB), tx_time (INTEGER). Indexes on (atom_hash), (tier).
- `capabilities` — same shape as atoms but a denormalized current-view for fast capability evaluation; populated by trigger or refreshed on capability supersession.
- `provenance` — atom_hash (FK), source_kind (TEXT: ingest_file, mcp_agent, federation_pull, fast_path), source_uri (TEXT), source_hash (BLOB).
- `entities` — entity_id (PK), canonical_label (TEXT, derived/computed from current atoms about the entity).
- `claims_fts` — SQLite FTS5 virtual table for prose-claim full-text search; populated by trigger from `atoms.envelope` payload.
- `federation_peers` — peer_id (PK, BLOB Ed25519 fingerprint), endpoint (TEXT), cert_fingerprint (BLOB), last_pull (INTEGER), bridge_capability (BLOB).
- `working_set` — projection_path (TEXT, PK), materialized_at (INTEGER), last_render_hash (BLOB), pinned (BOOL).

`atoms.envelope` stores the full canonical-JSON envelope (per ADR-017). The other columns are extracted indexable copies. The envelope is the source of truth; the columns exist for query efficiency.

Bitemporal queries follow the pattern: `WHERE entity_id = ? AND predicate = ? AND tx_time <= ? AND (valid_to IS NULL OR valid_to > ?) ORDER BY tx_time DESC`. The composite indexes make these point lookups in MVP graph sizes.

## Alternatives Considered

### Alternative 1: Separate SQLite database per predicate family

- **Description**: One database per family (`contacts.db`, `people.db`, `notes.db`) plus a `meta.db` for capabilities and provenance.
- **Pros**: Logical separation; smaller files; per-family backup.
- **Cons**: Cross-family queries (e.g., entity that has both contact and note atoms) require attached databases, complicating transactions and SQLCipher key management. Federation pulls atoms across all families and would have to dispatch by predicate. The benefit (separation) is not actually leveraged by any MVP feature.
- **Why rejected**: YAGNI. Single database is simpler and meets MVP scale.

### Alternative 2: Append-only event log + materialized views

- **Description**: All atoms append to a single journal table; materialized views (`current_atoms`, `current_capabilities`) refresh on insert via triggers or a background reconciler.
- **Pros**: Cleanest provenance story (the journal is the source of truth). Excellent for a Datomic-shaped substrate.
- **Cons**: View-refresh complexity (especially for supersession chains) is significant for MVP. Bitemporal queries against materialized views need careful trigger design or a reconciler with its own correctness story. Not a meaningful win at MVP graph sizes.
- **Why rejected**: Premature engineering. The normalized schema with `supersedes` and `tx_time` already preserves history; the journal-and-view pattern is reserved for Phase 3+ when scale demands it.

### Alternative 3: Application-layer AEAD per atom payload instead of SQLCipher

- **Description**: Store atom envelopes encrypted with ChaCha20-Poly1305 in `atoms.envelope`; the database file itself is plaintext. Indexable columns (entity_id, predicate, etc.) are unencrypted.
- **Pros**: Atoms can move between graphs as encrypted blobs without re-encrypting. No SQLCipher dependency.
- **Cons**: Indexable columns leak metadata (which entities and predicates exist, when, by whom) to anyone with file-system read access. The PRD's per-cohort DEK model targets *content*, not *metadata*; SQLCipher protects both in MVP. The indexable copies needed for fast queries cannot themselves be encrypted without breaking SQL.
- **Why rejected**: SQLCipher is the right granularity for MVP. Application-layer payload encryption is reserved for Phase 2+ when atoms move between cohort boundaries.

## Consequences

### Positive

- Single file is easy to back up, clone via git (the day-one-clone-and-collaborate scenario), and reason about.
- Bitemporal queries are fast at MVP scale via composite indexes.
- The canonical-JSON envelope in `atoms.envelope` is the single source of truth; indexable columns are derived and can be rebuilt if corrupted.
- SQLCipher integrates cleanly with the OS keychain DEK model.

### Negative

- Single-file write contention: only one writer at a time. The daemon serializes writes, which is acceptable for MVP scale but a future bottleneck.
- SQLCipher adds a build dependency; cross-compilation requires SQLCipher headers per target.

### Risks

- Schema migration as the substrate evolves. **Mitigation**: a `schema_version` table; daemon refuses to start against an unknown future version; migrations are forward-only and tested on a representative MVP graph.
- FTS5 index drift if envelope payload format changes. **Mitigation**: FTS5 is rebuildable from `atoms.envelope`; rebuild on schema migration.

## Implementation Notes

- All atom writes go through a single `AtomWriter` type in `ffs-core` that takes a canonical-JSON envelope, validates the signature, computes the BLAKE3 multihash, inserts into `atoms`, populates `classifications` and `provenance`, and triggers FTS5 indexing.
- Reads use prepared statements; the projection renderer composes a small set of typed queries rather than constructing SQL strings.
- Capability evaluation uses a fixed evaluator (per PRD § Capability as data) implemented as a Rust function; the `capabilities` table is the input set, the entity/predicate/tier the question.
- Backup is `cp ~/.ffs/store.db backup.db` plus the keychain DEK separately.
