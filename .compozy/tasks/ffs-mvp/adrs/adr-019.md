# ADR-019: Local IPC via Unix domain socket / Windows named pipe with JSON-RPC 2.0

## Status

Accepted

## Date

2026-05-05

## Context

The daemon hosts the substrate; the CLI, Obsidian plugin, and MCP server are local clients that read and write through it. They run on the same machine as the daemon. Three mechanisms compete:

- A local IPC primitive (Unix domain socket on Linux/macOS, named pipe on Windows).
- A localhost HTTP server bound to 127.0.0.1.
- gRPC over either of the above.

This boundary is consequential: the CLI uses it for every command; the plugin opens a long-lived connection; the MCP server delegates each MCP tool call to it. Latency, reliability, and security model all flow from the choice.

## Decision

The daemon exposes a single local-only RPC endpoint:

- On Linux/macOS: Unix domain socket at `~/.ffs/run/ffs.sock` with `0600` permissions.
- On Windows: named pipe at `\\.\pipe\ffs-<user_sid>`.

The wire protocol is JSON-RPC 2.0 (newline-delimited frames). Methods follow the verb-object pattern: `atom.get`, `atom.list`, `projection.render`, `path.list`, `capability.evaluate`, `ingest.submit`, `federation.peer.list`, `federation.peer.add`, `federation.pull`, `health.summary`, `predicate.inspect`. Notifications (server-to-client) carry filesystem-watch and substrate-change events: `event.atom.committed`, `event.projection.invalidated`, `event.fastpath.applied`.

Authentication is filesystem-permission-gated: only processes running as the substrate's user can open the socket. No bearer tokens, no TLS, no network exposure.

The MCP server, the CLI, and the Obsidian plugin all use the same JSON-RPC client built into `ffs-core`. Skills hosted by the daemon use the same RPC channel via stdio bridging (the daemon proxies skill calls through itself).

## Alternatives Considered

### Alternative 1: Localhost HTTP/2 with JSON over REST + SSE for events

- **Description**: Daemon binds to 127.0.0.1:<port>; clients use REST endpoints with SSE for events.
- **Pros**: Simpler client tooling (curl works). Most developers are fluent with HTTP.
- **Cons**: Loopback bind is an attack surface — any process on the machine, including malicious ones running as the same user, can connect. UDS / named pipes confine access to the user via filesystem permissions natively. Port allocation requires either a fixed port (collisions with other tools) or runtime discovery via a config file (race conditions). HTTP framing is heavier than JSON-RPC newline frames.
- **Why rejected**: The substrate is local-only by PRD commitment; the network surface should be zero. UDS satisfies that natively.

### Alternative 2: gRPC over UDS / named pipe

- **Description**: Same UDS / pipe transport but with gRPC framing and protobuf payloads.
- **Pros**: Strong typing. Mature streaming. Bidirectional flow control.
- **Cons**: Introduces protobuf as a build dependency for the Obsidian plugin (TS protobuf tooling is acceptable but adds friction). The atom envelope decision (canonical JSON, ADR-017) does not carry over to the API layer cleanly — gRPC would re-serialize the JSON payload, doubling encoding work. JSON-RPC's framing is sufficient for the daemon's traffic patterns.
- **Why rejected**: The complexity gain over JSON-RPC is not justified at MVP scale.

### Alternative 3: Each client uses a different mechanism

- **Description**: CLI uses UDS; plugin uses HTTP because TypeScript IPC is awkward; MCP server uses stdio.
- **Pros**: Each client uses what is most ergonomic for its language.
- **Cons**: Three RPC contracts to maintain, three sets of authentication and event delivery, three integration test surfaces. The daemon's API is already substantial; multiplying it by transport variants is not maintainable.
- **Why rejected**: One transport, one contract.

## Consequences

### Positive

- Zero network exposure. The daemon is unreachable from other machines without explicit federation transport (ADR-020).
- Filesystem-permission-gated authentication is the right granularity — the substrate is single-user.
- JSON-RPC 2.0 is well-specified, has mature client libraries in Rust, TypeScript, and Python, and is human-debuggable (`socat` + `jq` against the socket).
- Server-to-client notifications enable the plugin's optimistic-UI updates and the daemon's filesystem-watch event delivery without polling.

### Negative

- TypeScript clients (the Obsidian plugin) need a UDS-or-named-pipe wrapper. Node has `net.createConnection({ path })`; Windows requires `\\.\pipe\` paths. Both work but are slightly less ergonomic than HTTP.
- Browser-based developer tools cannot connect directly; an HTTP bridge would be needed. Out of scope for MVP.

### Risks

- The Obsidian plugin's UDS integration on Windows hits a quirk in Node's named-pipe handling. **Mitigation**: tested on the three OS targets in CI; fallback to spawning the CLI as a subprocess if direct IPC fails.
- A future need for remote access. **Mitigation**: the JSON-RPC contract is transport-agnostic; a Phase 2+ remote access mode could expose the same methods over mTLS HTTPS without a contract change.

## Implementation Notes

- Frame format: each JSON-RPC message followed by `\n` (line-delimited JSON).
- Concurrency: tokio's `UnixListener` / `NamedPipeServer` accepts connections; each connection runs on its own task; the daemon's request dispatcher is shared.
- Backpressure: the server may pause notifications when the client's event queue exceeds 1000 messages; the client can request a resync via `event.resync`.
- The CLI is a thin shim: parse argv → construct JSON-RPC request → send → format response → exit.
- The MCP server is a thin shim: receive MCP tool call → translate to JSON-RPC method → send → translate response to MCP result.
