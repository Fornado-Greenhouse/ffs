# ADR-023: Code-signed macOS binaries with shared `keychain-access-group`

## Status

Accepted

## Date

2026-06-13

## Context

Task_27 added the macOS Keychain integration: helper functions in `ffs-core::store::keyring` that wrap the `keyring` v3 crate, plus a daemon-side precedence chain (env-var → keychain → generate-and-warn). Unit tests passed; the `FFS_KEYRING_DISABLE` integration test passed. But the first live launchd deploy revealed three distinct daemon identities across three consecutive launchd-triggered boots:

```
boot 1 (launchd): owner=z3MD…  source=keychain
boot 2 (launchd): owner=zEoA…  source=keychain   ← different!
CLI (interactive): owner=zEXqe… source=keychain   ← also different!
```

Each boot wrote a fresh "keychain entry" that the next boot couldn't see, so each boot generated a new key, broke the SQLCipher DEK match against the existing `atoms.db`, and crashed.

Two compounding root causes:

1. **The keyring crate wasn't actually hitting macOS Keychain Services.** The workspace `Cargo.toml` declared `keyring = "3"` without specifying features. keyring v3 has *no default features*; on macOS without `apple-native` it silently falls back to its `mock` backend (an in-process HashMap that disappears on process exit). Every boot wrote to a fresh mock and "read" nothing on the next boot.

2. **Even with `apple-native` enabled, keyring v3 doesn't pass `kSecAttrAccessGroup`.** Its macOS backend calls `find_generic_password` / `set_generic_password` from `security-framework` without supplying an access group. Apple Keychain Services partitions generic-password entries by the writing binary's *code-signing identity* AND its *launch context*. So even after fixing the first bug, the launchd-spawned daemon (no terminal, runs as a daemon process) would not see entries written by the interactively-spawned CLI (terminal-attached, runs as a user process). Different launch contexts ⇒ different keychain partitions ⇒ same code, same binary, but invisible writes.

The fix is to (a) actually use the real macOS keychain backend, and (b) place every FFS binary in a shared, explicitly-named keychain access group so the partitioning doesn't matter.

## Decision

FFS adopts a **two-part fix**:

1. **Enable `keyring` v3's `apple-native` feature** on the workspace dep so the macOS backend actually reaches `Security/Security.h` instead of the mock. Also enable the equivalent native backends for Linux (`linux-native-sync-persistent`) and Windows (`windows-native`) so the crate's behavior is consistent across the matrix.

2. **Drop down to `security-framework` directly on macOS** in `ffs-core::store::keyring_macos`, calling `set_generic_password_options` / `generic_password` with an explicit `kSecAttrAccessGroup`. The access group is `3S9R9K2L38.com.ffs.shared` (project lead's Apple Developer Team ID + project namespace).

3. **Codesign every FFS binary** (`ffs`, `ffs-daemon`, `ffs-mcp`) with an `entitlements/ffs.entitlements.plist` declaring `keychain-access-groups = [3S9R9K2L38.com.ffs.shared]`. `scripts/codesign-macos.sh` is the local-dev tool; the GitHub Actions release workflow does the same with secrets-fed identity.

4. **Gate the keychain path on signed-binary detection at startup**. The daemon shells out to `codesign -d --entitlements - <self>` and looks for the access-group string in the output. When the running binary lacks the entitlement (unsigned dev build, missing Developer ID), the daemon logs a one-time warning and routes the keychain helpers down the same "disabled" path that `FFS_KEYRING_DISABLE=1` triggers — keeps the dev loop fast and avoids the cross-process chaos task_27's first deploy hit.

## Alternatives Considered

### Alternative 1 — Keep keyring v3 + accept partitioned writes

- **Description:** Enable `apple-native` only; don't set an access group; accept that the daemon and CLI see different entries.
- **Pros:** Smallest change. No `security-framework` direct dep.
- **Cons:** Doesn't actually solve the problem. The daemon would still generate a new key on every launchd boot and the substrate would still be unusable.
- **Why rejected:** Punts the bug.

### Alternative 2 — File-based shared secret instead of keychain

- **Description:** Skip the keychain entirely. Write the owner-key seed and DEK to `$FFS_DATA_DIR/run/keystore.json` mode `0600`.
- **Pros:** No keychain plumbing, no signing requirement, works everywhere identically.
- **Cons:** The whole point of the keychain integration is to put the secrets *somewhere other than disk*, where a file-system backup, accidental sync to a cloud drive, or git mishap can't carry them off. The Unix `chmod 0600` only protects against other local users; the keychain protects against the user's own missteps too. The PRD's "this is records about my life" framing means the user is going to back up `$FFS_DATA_DIR`; the keychain belongs outside that backup boundary.
- **Why rejected:** Defeats the security goal task_27 was scoped to meet.

### Alternative 3 — Submit a keyring v3 patch upstream

- **Description:** Add `with_access_group(group)` to `keyring::Entry` in the upstream crate, then use that from `ffs-core`.
- **Pros:** Doesn't fork the dep landscape. Other projects with the same need would benefit.
- **Cons:** Upstream timeline is unbounded. The FFS critical path (task_27 unblock) can't wait. Even if the patch lands quickly, the new API would be on a future minor version we'd have to wait to use.
- **Why rejected:** Time horizon. Worth doing as a parallel upstream contribution, but not as the FFS unblock.

## Consequences

### Positive

- The launchd-vs-interactive partitioning problem stops mattering: both contexts read the same access-group bucket.
- Signed-binary detection means the unsigned dev loop (running `cargo run`) keeps working without keychain entanglement — the gate logs a clear pointer to `scripts/codesign-macos.sh` and falls through to the env-var / generate path.
- The CI release workflow produces signed `.tar.gz` artifacts when the three signing secrets are configured. Forks without those secrets still get builds; they just get unsigned binaries that follow the env-var-or-generate fallback.

### Negative

- New runtime dep on `codesign` (a stock macOS tool — present on every macOS install with Command Line Tools). Without it, `is_signed_with_keychain_entitlement` returns false and the keychain path stays off.
- The signing requirement raises the floor for "production-shaped macOS install." A user building locally must either run `scripts/codesign-macos.sh` after `cargo build --release` or accept the unsigned/env-var path. The technical-friend-checklist now documents both routes.
- `security-framework` is added as a direct macOS-only dep on `ffs-core`. It was already a transitive dep via `keyring`'s `apple-native` feature; the direct add is small but it does increase the macOS build surface.

### Risks

- **Apple keychain partitioning rules change.** Apple occasionally tightens what entitlements a binary needs to read its own writes. The `keychain-access-groups` mechanism has been stable since macOS 10.10, but Apple's behavior under hardened runtime + notarization is an evolving target. **Mitigation:** the daemon's gate fails open to env-var, so a future Apple-side surprise produces a clear log message rather than data loss.
- **Notarization adds extra steps for distribution.** Codesigning is necessary but not sufficient for Gatekeeper to admit a downloaded `.dmg` past macOS 10.15. Notarization is not in this ADR's scope; it's a follow-up when FFS ships a public `.dmg`. The `--options runtime` flag in the codesign script is the hardened-runtime opt-in that notarization will require.

## Implementation Notes

The `keychain-access-groups` value must use the *project-namespaced* form `<TeamID>.<group>`. Apple uses that prefix to ensure groups are unique across developers — without it the codesign step fails with "access group is not in entitlements" or similar.

The runtime detection helper looks at the **merged stdout + stderr** of `codesign -d --entitlements - --xml`. Some macOS versions route the entitlement payload to stderr instead of stdout; we treat any substring match as a positive ID.

The daemon's `keychain_path_enabled()` composite check is `!FFS_KEYRING_DISABLE && is_signed_with_keychain_entitlement()`. The env-var path takes priority because tests need to be able to skip the keychain even on a signed binary.

### AMFI requires an embedded provisioning profile, not just signing

The above is necessary but not sufficient. Empirical finding (2026-06-14, macOS 26.2): a Developer-ID-signed, notarized, hardened-runtime, entitled binary that **doesn't carry an embedded provisioning profile** is silently `SIGKILL`ed by AMFI (kernel-level Apple Mobile File Integrity) before `main()` runs. Exit code 137. Zero stderr. No log entries the unprivileged side can see. Strip the entitlement → the same binary launches normally.

`keychain-access-groups` is a **restricted** entitlement (per [Apple TN3125](https://developer.apple.com/documentation/technotes/tn3125-inside-code-signing-provisioning-profiles)). Restricted entitlements have to be allowlisted by either Apple's own signing authority (App Store path — not us) or an **embedded provisioning profile** whose entitlements dict matches the binary's claims. AMFI verifies this at exec; mismatch → `EXC_CRASH (SIGKILL)`, before any user code runs.

Quinn "Eskimo" Apple DTS in [thread/685723](https://developer.apple.com/forums/thread/685723):

> "Other entitlements must be allowlisted by a provisioning profile, just like on iOS… the fact that the keychain-access-groups entitlement must be allowlisted by a profile means that other developers can't impersonate your app to steal its keychain items."

There is no `--preserve-metadata` / `--deep` workaround. The single supported path is to embed the profile.

### Provisioning profile lifecycle for Developer ID CLI binaries

Apple's portal now hides the old "Keychain Sharing" capability toggle on the App ID page — Eskimo, [thread/782084](https://developer.apple.com/forums/thread/782084), "every App ID automatically gains the Keychain Access capability… there is no Keychain Sharing capability because there's nothing for it to do." Authorization happens via the profile, not via App ID capabilities. The provisioning steps:

1. **App ID** at `developer.apple.com` → Identifiers → "+" → App IDs → App. Bundle ID `com.fornado.ffs` (or any bundle ID owned by team `3S9R9K2L38`). No capability checkboxes need to be touched.
2. **Profile** at `developer.apple.com` → Profiles → "+" → **Distribution** section → **Developer ID** → select the App ID + the Developer ID Application cert. Download → save as `embedded.provisionprofile`.
3. **Embed at build time.** Apple's documented path for non-`.app` Mach-O binaries is the linker section `__TEXT,__provisioning`, populated via `ld -sectcreate __TEXT __provisioning embedded.provisionprofile` ([Eskimo, thread/657917](https://developer.apple.com/forums/thread/657917)). For Rust, each binary crate has a `build.rs` that emits:
   ```rust
   println!("cargo:rustc-link-arg=-Wl,-sectcreate,__TEXT,__provisioning,<path>");
   ```
   when `FFS_PROVISIONING_PROFILE` is set. Builds without the env var are no-ops; cross-OS builds don't care.

`scripts/codesign-macos.sh` refuses to sign a binary that lacks the `__TEXT,__provisioning` section — surfaces the configuration error at sign time instead of letting it manifest as an inscrutable runtime SIGKILL.

### CI integration test path (skip-when-unsigned)

The integration test (`signed_daemon_produces_stable_keychain_identity_across_boots` in `tests/sqlite_persistence.rs`) honors an `FFS_SIGNED_DAEMON_BIN` env var pointing at a pre-built, pre-signed daemon binary. cargo / nextest rebuild the test-build daemon every test run and re-`adhoc`-sign it, blasting any Developer ID signature off — so the test can't use `env!("CARGO_BIN_EXE_ffs-daemon")` for the signed scenario. The recommended local invocation:

```sh
FFS_PROVISIONING_PROFILE="$PWD/secrets/embedded.provisionprofile" \
  cargo build --release
./scripts/codesign-macos.sh \
  target/release/ffs target/release/ffs-daemon target/release/ffs-mcp
# notarize via `xcrun notarytool submit … --wait`
export FFS_SIGNING_IDENTITY="Developer ID Application: …"
export FFS_SIGNED_DAEMON_BIN="$PWD/target/release/ffs-daemon"
cargo nextest run -p ffs-daemon --test sqlite_persistence \
  signed_daemon_produces_stable_keychain_identity_across_boots
```

Without `FFS_SIGNED_DAEMON_BIN`, the test falls through to `CARGO_BIN_EXE_ffs-daemon` (the cargo-managed adhoc-signed binary) and skips with `eprintln!` when `FFS_SIGNING_IDENTITY` is also unset. Contributor PRs without an Apple Developer Program enrollment keep CI green.

### Defense-in-depth on the .p12 password

Initial draft (commit `edaa2cf`) stored the throwaway `.p12` password on disk in `secrets/p12.password` alongside the `.p12` itself. Subsequent revision removed it: defense-in-depth says two secrets in one place is one secret. The `.p12` stays in `secrets/` as an emergency restore; the password lives only in the user's password manager + the `MACOS_SIGNING_CERT_PASSWORD` GitHub Actions secret.
