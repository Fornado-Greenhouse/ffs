# ADR-020: Federation transport — mTLS over HTTPS with pull-based sync

## Status

Accepted

## Date

2026-05-05

## Context

PRD non-goals state "MVP uses direct HTTPS for federation; ZTM is Phase 2." The remaining technical questions: how do peers authenticate to each other, how are atoms transferred, and is sync push- or pull-based?

ADR-007 commits MVP federation to bilateral pair-by-pair flows: each side authors capability atoms granting tier-scoped views; only authorized atoms cross. The transport layer must:

- Authenticate peers cryptographically using their substrate identities (ADR-018: Ed25519 keys).
- Carry atom envelopes (canonical JSON per ADR-017) efficiently.
- Honor capability checks at the source substrate, not the receiver.
- Work behind typical home-NAT setups without requiring a relay.
- Support immediate revocation (a revoked capability stops crossing within seconds).

## Decision

Federation runs over HTTPS with mutual TLS (mTLS), pull-based.

**Identity**: each substrate's federation endpoint presents a TLS certificate generated from its Ed25519 signing key. The certificate's subject CN is the multibase-encoded public key. Self-signed; trust is established by certificate-fingerprint pinning during the bilateral handshake (no PKI / CA chain).

**Bridge handshake**: out-of-band exchange (paste the peer's URL + fingerprint into the CLI / plugin) plus an in-band `bridge.establish` RPC that exchanges capability atoms, anchor heights, and supported predicate vocabularies. The handshake completes when both sides have authored each other's capability atoms locally.

**Transport endpoints** (on each substrate's HTTPS server):

- `POST /federation/v1/handshake` — bridge establishment.
- `GET /federation/v1/atoms?since=<tx_time>&capability=<cap_hash>` — capability-filtered pull of atoms after a watermark.
- `GET /federation/v1/atom/<multihash>` — individual atom fetch by hash (for reference resolution).
- `GET /federation/v1/projection/<path>` — capability-filtered projection rendering for `from/<peer>/` paths.
- `GET /federation/v1/intersection/<entity>` — capability-aware intersection check used by `intersection/with/<peer>/`.

**Sync model**: pull-based. The receiving peer's daemon polls each bridge on a heartbeat (default 60s) and on demand (when a user navigates `from/<peer>/`). Pulls carry the requesting peer's capability hash; the source substrate evaluates the capability against the request and returns only matching atoms above the watermark.

**Revocation**: when a peer revokes (supersedes the capability atom), subsequent pulls see the new capability and return nothing. The receiving peer's daemon detects an empty result against a watermark that previously returned atoms and unmounts the `from/<peer>/` view. Revocation propagation latency is bounded by heartbeat (default 60s; user-tunable to 10s if immediate revocation matters). For *immediate* revocation, the revoking peer can also `POST /federation/v1/revocation-notice` to peers it knows about; peers are not required to honor the push, but accepting it reduces propagation latency to seconds.

**NAT traversal**: out of scope for MVP transport. Users behind NAT use Tailscale, port-forward, or a community Cloudflare Tunnel-style fronting. The substrate does not embed a NAT-traversal layer in MVP.

## Alternatives Considered

### Alternative 1: Plain HTTPS with bearer-token capabilities

- **Description**: Standard TLS server (any cert, including self-signed pinned per-peer). Capability atoms act as bearer tokens. All payloads carry FFS atom signatures so transport-layer identity is secondary.
- **Pros**: Simpler than mTLS to set up. Works behind any TLS-terminating proxy. Identity verification is at the FFS layer where it logically belongs.
- **Cons**: Bearer tokens leak — if the capability is intercepted (TLS misconfiguration, compromised client), it can be replayed. mTLS binds the request to the peer's substrate key cryptographically; no replay window exists. The PRD's small-community trust model still benefits from binding identity at the transport.
- **Why rejected**: Marginal complexity gain; meaningful security improvement.

### Alternative 2: Push-based sync with relay fallback

- **Description**: Source substrate pushes new atoms to peers; a community relay holds atoms when peer is offline.
- **Pros**: Low latency for atoms-flow-during-edit cases. Works around offline peers.
- **Cons**: Push delivery requires reachability (NAT problem) or a relay (PRD's personal-hardware-only commitment objects to shared infrastructure in MVP). Push complicates capability evaluation: the source must be able to reach all entitled peers, which couples the source's availability to peers' availability.
- **Why rejected**: Pull-based is simpler, works around NAT, and matches the personal-hardware constraint. Push-based delivery is reserved for Phase 2+ when ZTM addresses reachability.

### Alternative 3: A2A as the federation wire format from MVP

- **Description**: Use A2A Agent Cards and the A2A wire format for FFS-to-FFS federation directly.
- **Pros**: Any A2A-aware client can talk to FFS. Maximum interop.
- **Cons**: A2A's semantics (agent capabilities, message routing) do not match FFS's bridge-contracts-and-atoms shape; mapping is lossy. ADR-008 explicitly commits to FFS-native between FFS substrates and A2A only at the agent-to-agent boundary. Rolling A2A into MVP federation is the wrong layering.
- **Why rejected**: Out of scope for MVP per ADR-008.

## Consequences

### Positive

- Cryptographically authenticated peer identity at the transport layer.
- No bearer-token replay window.
- Pull-based sync works around NAT for the majority of personal-hardware deployments where one side is reachable.
- Capability evaluation happens at the source on every pull — there is no "ship atoms then check capability" risk.
- Revocation latency is bounded by heartbeat and is configurable; opt-in revocation-notice push reduces it further.

### Negative

- mTLS setup is slightly more complex than bearer tokens. The CLI / plugin must guide the user through the fingerprint-pinning step.
- Pull-based sync means atoms authored at the source are not visible to the peer immediately — there is up to one heartbeat of lag for navigation flows that didn't trigger an on-demand pull.
- Self-signed certificates are unfamiliar to TLS-terminating proxies (Cloudflare, nginx) that expect CA-issued certs; users running behind such proxies need direct exposure or an mTLS-aware front.

### Risks

- A peer's certificate fingerprint changes (key rotation). **Mitigation**: a `bridge.rotate` flow lets a peer present an old-key signature over a new-key cert; the receiving peer accepts and updates the pin.
- An MVP user behind double-NAT cannot establish the federation transport. **Mitigation**: documentation recommends Tailscale (or any zero-config mesh VPN) for the small-community use case; this is a deployment concern, not a substrate concern.
- A capability is leaked via a misbehaving peer. **Mitigation**: capabilities are scoped (predicate, entity, classification, tier); leak blast radius is bounded by the scope. Revocation is fast.

## Implementation Notes

- HTTPS server: `axum` with `rustls`; client TLS verifier accepts only certificates whose fingerprint matches a registered peer.
- Certificate generation: `rcgen` produces an X.509 cert from the Ed25519 signing key (Ed25519 is supported in TLS 1.3 directly).
- Atom transfer: the atoms endpoint streams a JSON-Lines stream of canonical envelopes; the receiver verifies each signature and content-hash before accepting.
- Watermarks are `tx_time` integers per peer per capability; persisted in `federation_peers.last_pull` (ADR-016).
- NAT-friendliness in documentation: recommend Tailscale; document port-forward setup; note that one peer reachable is sufficient for bilateral pulls in both directions.
