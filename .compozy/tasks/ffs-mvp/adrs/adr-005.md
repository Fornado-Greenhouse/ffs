# ADR-005: Editor-agnostic working set materialization

## Status

Accepted

## Date

2026-05-05

## Context

The substrate's projection surface — paths like `~/.ffs/contacts/by-name/S/Sarah_Chen.md` — can be implemented in three ways:

- **Virtual mount only.** Projections never exist on disk; a kernel-level filesystem driver intercepts file access calls and renders projections on demand. (FUSE on Linux/Mac, Dokan on Windows, etc.)
- **Materialized working set.** A subset of projections — recently-touched, user-pinned — are written to disk as real files. The daemon manages them, refreshing on substrate changes.
- **Obsidian-only.** Projections appear only inside Obsidian via the plugin. Other tools (Notepad, vim, Explorer, grep) never see them.

The choice determines whether FFS is genuinely a file system or an Obsidian extension. The "what if someone opens it with Notepad?" test is the load-bearing scenario.

## Decision

FFS materializes a working set of projection files on disk, managed by the daemon. Any editor that opens files — Obsidian, Notepad, Vim, VS Code, TextEdit — can open them. Operating-system-level filesystem watchers detect edits to projection files; edits made through any tool route into the ingest path as correction notebook entries.

## Alternatives Considered

### Alternative 1: Virtual mount via filesystem driver

- **Description:** Install a kernel-level filesystem driver (FUSE/Dokan/CBFS) that intercepts all file access in `~/.ffs/`. Projections are rendered on demand for every read; they never exist as bytes on disk.
- **Pros:** Always-fresh projections. Strict semantic separation between substrate and surface. No working-set management needed.
- **Cons:** Requires kernel-level driver installation (admin rights on Windows, signed kernel modules on macOS, FUSE config on Linux). Files vanish when daemon crashes. Antivirus, indexing services, backup tools all behave unpredictably. On Windows specifically, kernel-level filesystem drivers are a known reliability minefield.
- **Why rejected:** The friction floor (mixed-capability community installable with light support from a technical friend, per Q4) rules out kernel-driver installation as a prerequisite. Reliability concerns compound the friction problem.

### Alternative 2: Obsidian-only access

- **Description:** Projections appear only via the Obsidian plugin. Other tools don't see them. The `~/.ffs/contacts/` folder doesn't exist on disk; navigation is plugin-mediated.
- **Pros:** Simpler implementation. No filesystem watching needed.
- **Cons:** Forces every user to use Obsidian. Cuts off CLI users, vim users, VS Code users, the Linux community generally. Conflicts with the both-audiences-first-class commitment (ADR-002) since developer-audience users often work outside Obsidian. Cuts off the integration with `ffs://` URL resolution from non-Obsidian contexts.
- **Why rejected:** Incompatible with developer audience and with the Linux community vision implicit in the `ffs://` URL scheme strategy.

### Alternative 3: Hybrid materialization with TTL eviction

- **Description:** Materialize projections on demand when a user navigates to them; keep them on disk for some TTL after access; evict back to "not materialized" when stale. Like Dropbox Smart Sync or OneDrive Files-On-Demand.
- **Pros:** Responsive on first access. Bounded disk usage.
- **Cons:** First-access latency is noticeable. Working-set policy is complex. Inherits some of Dropbox/OneDrive's UX nuances around "is this file actually here?" Edge cases when search indexers run on partially-materialized folders.
- **Why rejected:** Premature optimization for MVP. The simpler "materialize a working set, daemon keeps it current" approach is sufficient at v1 scale. The hybrid approach becomes attractive when graphs grow beyond what fits comfortably on disk; that's Phase 3+ territory.

## Consequences

### Positive

- FFS is genuinely a file system. The Notepad/Explorer scenario works. Linux users with shell pipelines work. Windows Search indexes the working-set files.
- No kernel driver requirement; installation is a normal-user activity.
- Editor choice is the user's. Obsidian is a rich surface; Notepad is the floor.
- The daemon's working-set management is a normal-user-process concern, simpler operationally than a kernel driver.

### Negative

- Working-set policy is an open question (see PRD § Open Questions). Wrong heuristics could materialize too much (disk pressure) or too little (search misses entities the user has).
- Edits made outside the daemon's awareness (e.g., direct `git pull` modifying files while daemon is down) need reconciliation. The daemon's restart logic must handle "files changed while I wasn't running."
- Filesystem watcher integration is OS-specific (ReadDirectoryChangesW on Windows, kqueue on macOS, inotify on Linux), which is real engineering work.

### Risks

- Working-set materialization may not scale to very large substrates. **Mitigation:** Phase 3+ adds the hybrid TTL-based approach when scale demands it; MVP is for personal-hardware sizes.
- Files-on-disk introduces cache-invalidation complexity. **Mitigation:** The "edits to projections become corrections in /ingest/" rule (already part of the architecture) avoids the worst case where users overwrite projections and the daemon has to reconcile.

## Implementation Notes

The MVP working-set heuristic is "most-recently-touched plus user-pinned." Refinement based on real use is Phase 2 (see Open Questions in PRD).

Editor-agnostic edit handling has two paths in MVP:

- **Fast-path** for trivial edits to the three MVP predicate types — single-line text-field edits, frontmatter value edits, and additive edits to designated sections. Detected via reverse-map annotations on predicate specs; authored as supersession atoms within ~200ms; projection re-renders to reflect the edit. (See ADR-014.)
- **Slow-path** for ambiguous edits — multi-paragraph restructuring, substantial deletions, edits spanning multiple atoms ambiguously, edits to federated projections, edits to projections rendered from predicates without reverse-map annotations. These route to the ingest folder as correction notebook entries; the scribe processes them; the user reviews and accepts in their daily health summary.

Both paths are MVP. Phase 2 adds the staged-edit pattern (explicit user-review surface for ambiguous cases beyond what the slow-path handles) and fast-path coverage for Phase 2 predicate types as the path library expands.
