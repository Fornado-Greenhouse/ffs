# ADR-015: Minimal FFS-specific daemon implemented in Rust

## Status

Accepted

## Date

2026-05-05

## Context

PRD § Open Questions explicitly defers the daemon implementation choice (OpenClaw skill set vs. minimal FFS-specific daemon) to implementation time. ADR-009 accepts both as architecturally supported but leaves the operational pick open. The TechSpec must commit so the build order, language stack, and interface contracts are well-defined.

The daemon owns a substantial set of responsibilities:

- Hosts the substrate's local API for the CLI, Obsidian plugin, and MCP server.
- Watches OS filesystem events (inotify on Linux, kqueue on macOS, ReadDirectoryChangesW on Windows) to detect projection-file edits.
- Runs the fast-path edit classifier and authors supersession atoms.
- Manages the working set of materialized projections.
- Coordinates federation (mTLS pull-based) with peer substrates.
- Hosts the FFS agents (scribe, librarian, auditor) packaged as `SKILL.md` artifacts.

The PRD also requires the `ffs` CLI as a single static binary per platform (Linux, macOS, Windows) and demands sub-second CLI response, sub-200ms folder navigation, sub-500ms projection rendering, and ~200ms fast-path edit acknowledgement.

## Decision

The MVP ships a minimal FFS-specific daemon written in Rust. The CLI and MCP server are also Rust binaries from the same workspace. The Obsidian plugin remains TypeScript. Agent skills (scribe, librarian, auditor) are Python files inside `SKILL.md`-shaped directories invoked by the daemon via subprocess + stdio.

Rationale:

- **Single static binary**: Rust produces fully static binaries with no runtime dependency, satisfying the PRD's CLI distribution requirement directly.
- **Filesystem watchers**: the `notify` crate wraps inotify/kqueue/ReadDirectoryChangesW behind a uniform API.
- **Cryptography**: `ed25519-dalek`, `chacha20poly1305`, and `blake3` are mature, audited Rust crates that match the chosen primitives (ADR-018).
- **Latency control**: no GC pauses, predictable memory; the daemon can hold long-running connections without jitter.
- **Concurrency**: `tokio` for async I/O on local sockets and HTTPS; `rusqlite` (or `sqlx`) for the atom store.
- **Performance budget**: the fast-path's 200ms target needs predictable performance from filesystem-event to atom write to projection re-render.

The agent host is in-process: a `skills/` registry the daemon reads at startup; each skill runs in a child process the daemon spawns. The daemon exposes a small skill-side API (read substrate, write atoms via ingest) over the same JSON-RPC channel (ADR-019). This is "minimal" in the sense that it does not implement the full claw pattern (multi-channel routing, model-agnostic LLM mediation, identity federation across hosts); skills compatible with OpenClaw remain runnable inside OpenClaw if a user prefers, because the `SKILL.md` and `definition.atom.json` artifacts are claw-shape conformant per ADR-009.

## Alternatives Considered

### Alternative 1: OpenClaw integration as primary host

- **Description**: FFS ships only as a set of `SKILL.md` skills loaded by an existing OpenClaw installation. The "daemon" is OpenClaw; FFS contributes substrate skills and a thin substrate library.
- **Pros**: Leverages claw ecosystem. Multi-channel routing comes for free. Less infrastructure code.
- **Cons**: OpenClaw skill-system maturity through MVP timeline is uncertain. Forces OpenClaw as a hard prerequisite, contradicting the friction-floor commitment (technical-friend onboarding, light support). Filesystem-watcher integration, fast-path edit detection, working-set management, and federation transport are not OpenClaw concerns and would need to be bolted on as either skills or external companions.
- **Why rejected**: Not the right shape for MVP. The substrate's distinctive infrastructure (FS watchers, fast-path, federation) does not map cleanly to claw skills. OpenClaw integration remains an *optional* path because skills are claw-shape conformant; it is not the *primary* host.

### Alternative 2: Both, with OpenClaw as optional secondary

- **Description**: Ship the minimal FFS daemon as primary; also provide skill manifests so the agents can run inside OpenClaw for users who already have it.
- **Pros**: Most flexible. Honors ADR-009 commitment to claw-pattern compatibility.
- **Cons**: Two integration paths to test, document, and support during MVP. Splits attention across two installation experiences. The agent skills are already claw-shape conformant, so the secondary path exists implicitly without needing MVP-level investment.
- **Why rejected**: Operational scope creep. The minimal FFS daemon is the supported MVP path; the OpenClaw-host path becomes a Phase 2 or community-contributed integration, not an MVP commitment.

### Alternative 3: Go for daemon/CLI/MCP

- **Description**: Same minimal-daemon architecture but in Go. Go produces static binaries, has `fsnotify`, decent crypto stdlib, simpler than Rust.
- **Pros**: Easier learning curve. Faster compile times. Mature tooling.
- **Cons**: GC pauses (small but real) and larger binaries. The crypto ecosystem for Ed25519/ChaCha20-Poly1305/BLAKE3 in Rust is more mature than in Go (BLAKE3 specifically). Memory and latency control matters for the fast-path latency target.
- **Why rejected**: Marginal. Rust's predictable latency and crate ecosystem for crypto and FS watchers tipped the choice. Go would have been an acceptable second.

### Alternative 4: TypeScript/Node or Python for daemon

- **Description**: Use a high-level language for the daemon to share more code with the plugin or skills.
- **Pros**: Single-language ergonomic story.
- **Cons**: Neither has a credible single-static-binary story for the CLI. Filesystem-watcher portability is OK in Node but not great. Python's GIL and PyInstaller fragility make it the wrong language for a substrate-layer daemon serving sub-200ms latency targets.
- **Why rejected**: Mismatched runtime constraints.

## Consequences

### Positive

- Single static binary for the CLI on each platform, satisfying PRD distribution requirements directly.
- Predictable latency for fast-path edit detection and atom authoring.
- Mature crate ecosystem covering filesystem watchers, crypto, SQLite, async I/O, and HTTP.
- The daemon, CLI, and MCP server share a Rust workspace; common types (atom envelope, capability evaluator) are reused, not duplicated.
- Skills remain Python and claw-conformant, preserving Phase 2 OpenClaw-host portability.

### Negative

- Rust learning curve for contributors who do not already know it.
- Compile times are slower than Go or interpreted languages.
- Heterogeneous stack (Rust + TypeScript + Python) means three toolchains in CI.

### Risks

- The fast-path classifier's complexity grows beyond what is comfortable to maintain in Rust. **Mitigation**: the classifier is bounded to three predicate types and three edit categories (per ADR-014); reverse-map rules are declarative TOML (ADR-021), not Rust code.
- A future need for OpenClaw-host parity becomes urgent. **Mitigation**: skills are already claw-shape conformant; promoting the OpenClaw integration to primary or secondary in Phase 2 is incremental, not a rewrite.

## Implementation Notes

- Workspace layout: `crates/ffs-core` (atom model, capability evaluator, store), `crates/ffs-daemon` (binary), `crates/ffs-cli` (binary), `crates/ffs-mcp` (binary), `crates/ffs-federation` (HTTPS + mTLS), `crates/ffs-fastpath` (reverse-map classifier), `crates/ffs-skills-host` (subprocess agent host).
- Async runtime: `tokio` with multi-thread scheduler.
- HTTPS server/client: `rustls` + `hyper` or `reqwest`.
- SQLite: `rusqlite` with the `bundled-sqlcipher` feature for at-rest encryption.
- Filesystem watcher: `notify` crate with debounced events.
- Logging: `tracing` with structured JSON output for the auditor's daily report.
