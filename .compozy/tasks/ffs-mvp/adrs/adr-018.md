# ADR-018: Cryptographic primitives — Ed25519, ChaCha20-Poly1305, BLAKE3

## Status

Accepted

## Date

2026-05-05

## Context

The PRD commits to: signed atoms; AEAD encryption at rest with per-cohort DEKs managed via the OS keychain; content addressing via multihash for forward compatibility. The specific algorithms are unspecified. Atoms signed with one algorithm cannot be verified by an FFS instance using a different algorithm; this is therefore a long-lived interoperability contract, not a private implementation detail.

The choice space:

- **Signing**: Ed25519 (modern, fast, 32-byte keys, 64-byte signatures, deterministic) vs. ECDSA-P256 (FIPS-blessed, more legal-friendly) vs. Ed448 (higher security margin, slower).
- **AEAD**: ChaCha20-Poly1305 (constant-time on every CPU, no hardware needed) vs. AES-256-GCM (hardware-accelerated where AES-NI exists) vs. XChaCha20-Poly1305 (192-bit nonce, safer for random nonces).
- **Content hash**: BLAKE3 (fastest, parallelizable) vs. SHA-256 (most universal) vs. SHA-512 (higher margin).

Federation transport (mTLS, ADR-020) uses a separate but related set of primitives.

## Decision

The MVP cryptographic suite:

- **Signing**: Ed25519 (RFC 8032).
- **AEAD encryption at rest**: ChaCha20-Poly1305 (RFC 8439). Nonces are 96-bit, generated by counter (per-key strict counter, persisted with the DEK metadata).
- **Content hash**: BLAKE3 (32-byte digest), wrapped in multihash with codec `0x1e`.
- **Key derivation**: HKDF-SHA-256 for deriving sub-keys from the master DEK.

Author identity is the Ed25519 public key, encoded as multibase (base58btc, prefix `z`) for human-readable representation in atoms, capabilities, federation peer IDs, and the `ffs://` URL author parameters.

The AEAD scope: SQLCipher uses ChaCha20-Poly1305 internally (via its `cipher_chacha20` page cipher) for per-database encryption. Application-layer AEAD on individual atom payloads is reserved for Phase 2+.

## Alternatives Considered

### Alternative 1: AES-256-GCM AEAD with SHA-256 content hash

- **Description**: AES-256-GCM for encryption, SHA-256 for content addressing. Most conservative, FIPS-aligned, hardware-accelerated where AES-NI is present.
- **Pros**: Universal availability. AES-NI hardware acceleration on most modern CPUs. SHA-256 is in every standard library.
- **Cons**: GCM has notorious nonce-misuse footguns; a single nonce reuse breaks confidentiality and authenticity catastrophically. Without AES-NI hardware (older ARM, embedded), AES is slow and timing-attack-prone. SHA-256 is fine but BLAKE3 is 3-10x faster for content hashing of MVP-sized payloads.
- **Why rejected**: ChaCha20-Poly1305 is constant-time on every platform without hardware support and has strictly less risk of nonce-misuse failure modes than GCM. BLAKE3 is faster and parallelizable.

### Alternative 2: XChaCha20-Poly1305 AEAD with SHA-256 content hash

- **Description**: XChaCha20-Poly1305 (192-bit nonce) for safer random nonces; SHA-256 for content hash compatibility.
- **Pros**: 192-bit nonces eliminate practical concerns about nonce reuse with random generation. SHA-256 universal.
- **Cons**: XChaCha20-Poly1305 is less standardized than ChaCha20-Poly1305 (a CFRG draft, not an RFC). Marginal benefit for MVP since DEK nonce management uses a strict counter, not random nonces.
- **Why rejected**: ChaCha20-Poly1305 with counter nonces is the more widely standardized choice; XChaCha20's marginal safety wins do not apply.

### Alternative 3: ECDSA-P256 signing for FIPS alignment

- **Description**: ECDSA over NIST P-256 instead of Ed25519. FIPS 186-4 approved.
- **Pros**: FIPS compliance becomes possible if MVP users care.
- **Cons**: ECDSA signatures are non-deterministic (depend on a random nonce); a bad RNG can leak the private key (the Sony PS3 incident). Implementation is more error-prone. FIPS alignment is not an MVP target (PRD § Compliance defers regulatory regimes).
- **Why rejected**: Compliance is not an MVP success criterion. Ed25519's deterministic signatures and simpler implementation reduce risk.

## Consequences

### Positive

- Ed25519 is supported by every modern crypto library; verifiers can be implemented anywhere with minimal code.
- ChaCha20-Poly1305 has constant-time, side-channel-resistant implementations everywhere.
- BLAKE3 hashing is fast enough that atom canonicalization-and-hashing is not a bottleneck even on the fast-path's 200ms budget.
- The combined suite is the modern default across cryptographic libraries; common audited implementations (`ed25519-dalek`, `chacha20poly1305`, `blake3` in Rust) have wide review.

### Negative

- BLAKE3 is less universal than SHA-256; some downstream tools may not support it. Multihash codec wrapping mitigates this — the algorithm ID is part of the address, so the substrate can adopt SHA-256 alongside BLAKE3 if a federation peer requires it.
- FIPS compliance is harder. MVP non-target; flagged for later.

### Risks

- A future cryptographic break in Ed25519, ChaCha20-Poly1305, or BLAKE3. **Mitigation**: multihash content addressing provides forward migration for the hash; capabilities can be re-issued under a new signing algorithm; AEAD can rekey the database. The substrate is not committed to one algorithm forever.
- A nonce-counter persistence bug causes nonce reuse. **Mitigation**: nonce counter is fsync'd to keychain metadata before use; a recovery path on restart re-derives the next nonce conservatively.

## Implementation Notes

- Rust crates: `ed25519-dalek` v2 for signing, `chacha20poly1305` for AEAD, `blake3` for hashing, `multihash` for the codec wrapper, `hkdf` + `sha2` for key derivation.
- Keychain integration: the `keyring` crate (cross-platform: macOS Keychain, Windows Credential Store, Linux Secret Service / kwallet).
- DEK generation: 32 random bytes from the OS CSPRNG; stored in keychain under service name `ffs-dek-<cohort_id>`.
- Author signing keys: 32 random bytes from the OS CSPRNG; private key stored in keychain under `ffs-author-<author_id>`; public key embedded in atoms.
- Federation transport uses the same Ed25519 key as the substrate's signing identity; the mTLS certificate is generated from it (ADR-020).
