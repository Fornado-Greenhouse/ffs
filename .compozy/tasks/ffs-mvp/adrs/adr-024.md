# ADR-024: Windows ACL hardening at install time via `icacls`, not in the daemon

## Status

Accepted

## Date

2026-06-11

## Context

The Unix daemon binary applies `chmod 0o700` to `$FFS_DATA_DIR/run/` at startup (`crates/ffs-daemon/src/main.rs`) so a malicious local process can't swap the UDS file out from under us between bind and accept. That's a defense-in-depth check sitting on top of the OS's per-user home-directory protections.

Windows has no `chmod`. The closest equivalent is to write a DACL granting Full Control to the current user and nothing to anyone else. Two implementation sites were considered for task_34:

- **Option A — daemon-side.** Call `SetNamedSecurityInfoW` from `crates/ffs-daemon/src/main.rs` via `windows-sys`. Pros: enforcement runs every daemon boot, even if the user side-stepped the installer. Cons: pulls a large platform crate into the daemon for what is effectively a setup-time concern; the ACL code lives in main.rs forever and needs to be maintained alongside the Unix path.

- **Option B — installer-side.** Have `installer/install.ps1` invoke `icacls` once at install time to lock down `$FFS_DATA_DIR` (and its `run/` subdirectory) to the current user. Pros: matches the Unix story exactly (the *installer* runs `chmod 0o700`, not the daemon — `installer/install.sh` already does this). Keeps the daemon binary platform-agnostic. Cons: someone who deletes the ACL between install and boot gets an unhardened directory until the next install.

## Decision

Task_34 uses **Option B (installer-side `icacls`)**.

`installer/install.ps1` invokes `icacls "$DataDir" /inheritance:r /grant:r "$env:USERNAME:(OI)(CI)F"` on the data root and the `run/` subdirectory after creating them. The daemon's `main.rs` leaves the existing `#[cfg(unix)]` `chmod 0o700` block alone and adds no Windows counterpart.

## Alternatives Considered

### Option A — daemon-side `SetNamedSecurityInfoW`

- **Description:** The daemon binary calls `windows-sys` ACL APIs to harden `$FFS_DATA_DIR/run/` at every startup.
- **Pros:** Re-enforces hardening on every boot. Survives an ACL deletion between install and boot.
- **Cons:** Adds a meaningful Windows-only dependency (`windows-sys` Win32::Security feature set, ~300 KB of generated bindings) to the daemon binary for what is fundamentally a setup-time concern. Asymmetric with the Unix story where the chmod lives in the *installer*, not the daemon — Unix's `installer/install.sh` already does `chmod 0o700 "$DATA_DIR"`. The daemon-side Unix chmod at `main.rs:135` is an additional belt-and-braces guard specifically on the socket's parent dir, not the data dir.
- **Why rejected:** The dep-weight + complexity asymmetry isn't worth re-enforcement on every boot. A user who manually clears the ACL has bigger problems and will see the next install fix it.

### Option C — neither (rely on `$LOCALAPPDATA` / `$USERPROFILE` defaults)

- **Description:** Skip explicit ACL hardening; trust that Windows' default ACL on `%USERPROFILE%\.ffs\` (inherited from the user profile) already restricts to the current user.
- **Pros:** Zero installer code, zero daemon code, no platform deps.
- **Cons:** Default ACLs on `%USERPROFILE%` may inherit broader rights from parent folders depending on Group Policy / domain configuration / OEM image. Defense-in-depth is exactly the place to not rely on environmental defaults — that's why the Unix daemon does `chmod 0o700` *on top of* the home-directory default.
- **Why rejected:** Equivalent posture to the Unix story is the goal; "trust the default" doesn't meet it.

## Consequences

### Positive

- Daemon binary stays platform-agnostic. No `windows-sys` dep, no `#[cfg(windows)]` ACL block in `main.rs`.
- Story matches Unix: the installer hardens the data dir; the daemon trusts that hardening at runtime.
- `icacls` is a stock Windows command — no PowerShell module dependency, works on every supported Windows since 7.

### Negative

- An adversary who clears the ACL between install and daemon boot gets an unhardened state until the next install runs. The mitigation is "users who alter their data dir's ACL get what they ask for."

### Risks

- **`icacls` semantics edge cases.** `icacls` flag combinations are notoriously fiddly; the chosen `/inheritance:r /grant:r "<user>:(OI)(CI)F"` form has the specific effect of "remove inherited permissions, grant the current user Full Control with Object Inherit + Container Inherit." Verified via the `icacls "$Dir"` output post-install — the install.ps1 step is followed by a verification call.
- **Service vs. interactive identity.** If the user later registers the daemon as a Windows service rather than the at-logon scheduled task that install.ps1 wires up, the running identity differs from `$env:USERNAME`. That's out of scope for MVP (the installer uses the scheduled-task path) but should be revisited if a service installer lands.

## Implementation Notes

The `icacls` invocation lives in `installer/install.ps1`:

```powershell
function Harden-Acl($path) {
    if ($DryRun) {
        Write-Host "+ icacls $path /inheritance:r /grant:r `"$env:USERNAME`":(OI)(CI)F"
    } else {
        & icacls $path /inheritance:r /grant:r "$($env:USERNAME):(OI)(CI)F" | Out-Null
    }
}

Harden-Acl $DataDir
Harden-Acl (Join-Path $DataDir 'run')
```

The flags decompose as:

- `/inheritance:r` — remove inherited ACEs from the target. Without this, ACEs from `$env:USERPROFILE` (or further up if domain-joined) propagate.
- `/grant:r "user:(OI)(CI)F"` — replace any existing ACE for `user` with `Full Control`, applied to this folder, subfolders (Container Inherit), and files within (Object Inherit).
