# ADR-010: MCP server deferred to Phase 2

## Status

Superseded by [ADR-013](adr-013.md)

## Date

2026-05-05

## Note

This ADR was superseded the same day it was authored. The original deferral reasoning underweighted that the home-claw absorption scenario (one of the three MVP scenarios per ADR-004) requires bidirectional MCP for full validation: agents writing to the ingest folder is half the story, agents reading substrate state via MCP is the other half. ADR-013 captures the corrected decision.

The original ADR-010 content follows for historical record.

---

## Context

The FFS-MCP server exposes substrate operations (query, projection rendering, atom authoring with capability checks, `ffs://` URL resolution, predicate inspection, audit query) as MCP tools. Any MCP-aware AI agent — Claude, ChatGPT, Gemini, framework-agnostic agents — can read and write the substrate without bespoke integration once it ships.

The MCP server is the highest-leverage piece of FFS surface area for ecosystem reach. It's also a thin wrapper over the substrate's existing API; the substantial work is the substrate itself, not the MCP layer.

The both-audiences-first-class commitment (ADR-002) creates pressure to ship the MCP server in MVP — it's the developer-audience surface that scales. The Substrate-First MVP commitment (ADR-003) creates pressure to defer it — the MVP success criterion is small-community validation, not broad developer ecosystem reach.

## Decision

The FFS-MCP server is Phase 2, not MVP. MVP developer audience is served through the local API and the `ffs` CLI with `ffs://` URL resolution. Phase 2 adds the MCP server as a pre-committed deliverable.

## Alternatives Considered

### Alternative 1: Ship MCP server in MVP

- **Description:** Include the FFS-MCP server in MVP, alongside the CLI. Both serve the developer audience.
- **Pros:** Maximum developer-audience reach in MVP. Any MCP-aware agent becomes FFS-aware.
- **Cons:** The success criterion (small-community validation) is not advanced by MCP server work. The CLI is sufficient for developers writing applications against FFS in MVP. The MCP server is leverage for the *next* wave of adoption, not the first.
- **Why rejected:** Premature optimization for adoption that hasn't been earned. Validate MVP success before investing in ecosystem leverage.

### Alternative 2: Ship MCP server in MVP, defer CLI

- **Description:** Use MCP as the primary developer surface, skip the CLI.
- **Pros:** Sharper standards-citizenship message.
- **Cons:** Cuts off shell-pipeline use, the Linux community's natural addressing idiom, and `ffs://` URL resolution as a stable primitive. The CLI is part of the universal addressing story (ADR-006); MCP is for AI agents. They're not substitutes.
- **Why rejected:** The CLI and MCP server serve different purposes. Cutting the CLI cuts off Linux/Windows command-line workflows; cutting the MCP server is a more contained deferral.

## Consequences

### Positive

- MVP scope is sized to validate the success criterion. MCP server work doesn't push out the small-community validation timeline.
- Phase 2 delivers MCP server work after MVP feedback informs what tools the server should expose. The shape of the server's tools can be informed by what the small community actually does.
- The CLI's `ffs://` URL resolution is the developer-audience surface in MVP; that's a stable foundation the MCP server builds on in Phase 2.

### Negative

- Developers evaluating FFS in MVP must work through the CLI and local API rather than through MCP. For some developers, especially those building MCP-native applications, this is friction.
- Ecosystem leverage that the MCP server provides is delayed.

### Risks

- A competing substrate ships an MCP server first and captures developer mindshare. **Mitigation:** The substrate's distinctive value (records, federation, capability) is independent of MCP server shipping. Developers who care about substrate semantics will find FFS regardless of MCP timing.
- Phase 2 MCP server work surfaces complexities not anticipated. **Mitigation:** The thin-wrapper-over-existing-API model means the MCP server is a translation layer. Substrate semantics are settled in MVP; MCP layer work is bounded.

## Implementation Notes

The Phase 2 FFS-MCP server exposes these tools:

- `ffs_query` — query the substrate by entity, predicate, time range; capability-checked.
- `ffs_render_projection` — render a projection at a path.
- `ffs_resolve_url` — resolve an `ffs://` URL.
- `ffs_author_atom` — propose an atom into the ingest path with provenance; capability-checked.
- `ffs_inspect_predicate` — return a predicate spec for an agent to understand the vocabulary.
- `ffs_audit_query` — run audit queries (recent atoms, supersessions, classifications).

The exact tool surface is open; the small community's MVP usage will inform what's most needed.
