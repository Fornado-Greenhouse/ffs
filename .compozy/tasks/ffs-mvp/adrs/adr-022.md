# ADR-022: Substrate-is-vault â€” `$FFS_DATA_DIR` is the Obsidian vault root

## Status

Accepted

## Date

2026-06-02

## Context

The installer landed in task_22 with a `--vault <path>` flag that accepted any directory and copied the Obsidian plugin into `<vault>/.obsidian/plugins/ffs/`. The materializer (task_25) writes projection files to `$FFS_DATA_DIR/contacts/by-name/...`, `$FFS_DATA_DIR/notes/by-name/...`, etc. When the user passed `--vault /some/other/path`, the materializer wrote into `$FFS_DATA_DIR` (default `~/.ffs/`) and the user's Obsidian vault â€” which was elsewhere â€” appeared empty.

This was discovered during the post-task_26 first-use rehearsal: the substrate was working end-to-end, atoms were being signed and materialized, but the user's Obsidian file explorer (rooted at `~/Library/Mobile Documents/iCloud~md~obsidian/Documents/ffs-test/`) had nothing in it because materialized contacts landed at `~/.ffs/contacts/by-name/S/Sara_Chen.md`.

The original PRD and ARCHITECTURE.md always assumed `~/.ffs/` *was* the vault â€” the path-library structure is opinionated about that. The installer's `--vault` flag was inherited unreflectively from a generic "Obsidian plugin installer" mental model and never reconciled with the substrate's mental model.

## Decision

`$FFS_DATA_DIR` (default `~/.ffs/`) is canonically the Obsidian vault root. The installer seeds `$FFS_DATA_DIR/.obsidian/plugins/ffs/` directly. The user's first-time Obsidian step is **Open folder as vault â†’ `~/.ffs/`** â€” there is no separate vault to choose or configure.

The installer's `--vault <path>` flag stays accepted for backwards compatibility, but supplying a path different from `$FFS_DATA_DIR` triggers a warning to stderr/stdout explaining that the materializer writes to `$FFS_DATA_DIR` and an external vault will appear empty.

## Alternatives Considered

### Alternative 1: Configurable vault path that the materializer writes to

- **Description:** Make `$FFS_VAULT_DIR` separate from `$FFS_DATA_DIR`. The materializer writes projections to `$FFS_VAULT_DIR/contacts/by-name/...`; substrate state (atoms.db, ingest/, skills/, run/, log/) stays at `$FFS_DATA_DIR`.
- **Pros:** Respects users who already have an Obsidian vault and want to keep using it.
- **Cons:** Doubles the configuration surface. Splits the path-library â€” `ingest/` lives in one root and `contacts/` lives in another, even though they're conceptually parts of the same substrate. The fast-path watcher (task_09) becomes harder: it needs to watch two roots and reconcile cross-root events. The Obsidian plugin, the CLI, and the federation pull path all need to know the split.
- **Why rejected:** YAGNI. Federation, not iCloud sync or external vaults, is FFS's answer to "use this knowledge from multiple devices."

### Alternative 2: Symlink-based externalization

- **Description:** `$FFS_DATA_DIR` stays the single substrate root, but the installer can symlink `<external-vault>/contacts â†’ ~/.ffs/contacts`, same for `ingest/`, `notes/`, etc.
- **Pros:** Substrate stays unified; the user can keep their existing vault.
- **Cons:** Symlinks are fragile across platforms (iCloud sync resolves them inconsistently; Windows handles them differently; future federation pulls may walk through them in unexpected ways). The fast-path watcher's canonicalization (`std::fs::canonicalize`) already had to be added for `notify::PollWatcher` on macOS; symlinks would compound that.
- **Why rejected:** Real symlink semantics across three OSes is a maintenance liability that doesn't earn its place in MVP.

### Alternative 3: Status quo â€” installer requires explicit `--vault`

- **Description:** Keep the original task_22 design. User picks a vault path explicitly; installer warns when omitted; user is responsible for understanding that the vault path and the substrate path must match.
- **Pros:** No code changes.
- **Cons:** The user IS the one who hit the failure mode that prompted this ADR â€” the architecture didn't surface the constraint until it broke their experience. "User responsible for understanding" isn't a design; it's an admission that the design hasn't decided.
- **Why rejected:** The first-use rehearsal proved the failure mode is too easy to fall into. Picking a sensible default removes the trap.

## Consequences

### Positive

- One root path, one mental model: "FFS owns `~/.ffs/`. To use it, open `~/.ffs/` in Obsidian."
- The installer Just Works with no `--vault` argument â€” the friction floor for technical-friend onboarding drops by one decision.
- The materializer's path resolution, the fast-path watcher's root, and the Obsidian plugin's expected vault all point at the same directory. No cross-root reconciliation logic.
- `~/.ffs/` is not iCloud-synced by default (it lives directly under `$HOME`), which sidesteps the FS-watcher-vs-cloud-sync friction the troubleshooting guide already warns about. Federation handles cross-device data movement more cleanly than iCloud sync would, since FFS atoms are signed and content-addressed.

### Negative

- Users with strong "I already have an Obsidian vault and want FFS to live inside it" preferences must either (a) accept the warning when they pass `--vault`, or (b) wait for a future symlink-or-mirror feature (out of MVP scope).
- The `~/.ffs/.obsidian/` directory now contains Obsidian's per-vault state (workspace.json, etc.) alongside the substrate. It's organizationally clean â€” the `.obsidian/` directory is owned by Obsidian, and the substrate ignores it â€” but it adds visible structure under `~/.ffs/` that previously wasn't there.

### Risks

- **A user who runs the installer with an explicit `--vault` may not notice the warning** and end up with an empty external vault while their substrate populates `~/.ffs/`. **Mitigation**: the warning is repeated three times in the installer output (`WARN` lines on consecutive lines), the troubleshooting guide gets an explicit entry for it, and the post-task_30 docs no longer mention `--vault` outside of the migration section.

## Implementation Notes

- `installer/install.sh::install_obsidian_plugin()` defaults `VAULT_PATH="$DATA_DIR"` when the flag is unset.
- `installer/install.ps1` does the same for Windows.
- The `installer_layout` regression test (`crates/ffs-daemon/tests/installer_layout.rs`) covers the new layout: no `--vault`, `--vault $FFS_DATA_DIR`, and `--vault /external/path` all produce the right files in the right places (or warn, in the external case).
- `docs/onboarding/first-use-guide.md` opens with "Step 0: open `~/.ffs/` as your Obsidian vault."
- `docs/onboarding/technical-friend-checklist.md` includes a migration section for users who installed before this change.
