# ADR-013: MCP server in MVP

## Status

Accepted (supersedes [ADR-010](adr-010.md))

## Date

2026-05-05

## Context

ADR-010 deferred the FFS-MCP server to Phase 2, reasoning that the small-community success criterion is advanced by personal federation rather than by ecosystem-scale MCP integration, and that the CLI plus `ffs://` URL resolution is sufficient developer surface for MVP.

That reasoning underweighted a specific concern. The home-claw absorption scenario is one of the three MVP scenarios (per ADR-004): power users running many AI agents on personal hardware, with agents writing markdown to the substrate for absorption. Writing to ingest via the filesystem is one half of that scenario; reading substrate state via MCP is the other half.

Without an MCP server in MVP:

- AI agents can write markdown to the ingest folder but cannot query the substrate. They cannot ask "what are my recent decisions about the Acme deal?" or "what contacts do I have in tech?" before they generate output.
- The home-claw absorption scenario validates the substrate's *write* path but not its *read* path for MCP-aware agents.
- Developer-audience evaluation depends on the substrate-API or CLI rather than on the standardized agent integration protocol the rest of the ecosystem uses.

The MCP server is also a thin wrapper over the substrate's existing API. The substantial work is the substrate, which MVP ships anyway. Six MCP tools with capability checks at the boundary is on the order of days to weeks of implementation, not months.

## Decision

The FFS-MCP server is MVP, not Phase 2. The MVP server exposes six tools:

- **`ffs_query`** — query the substrate by entity, predicate, or time range; capability-checked.
- **`ffs_render_projection`** — render a projection at a path; equivalent to `ffs cat` over MCP.
- **`ffs_resolve_url`** — resolve an `ffs://` URL.
- **`ffs_author_atom`** — propose an atom into the ingest path with provenance; capability-checked at signing.
- **`ffs_inspect_predicate`** — return a predicate spec so an agent can understand the substrate's vocabulary.
- **`ffs_audit_query`** — run audit queries (recent atoms, supersessions, classifications, capability events).

Capability checks fire at the MCP boundary; an agent receives only what its identity is entitled to. An agent attempting to author an atom outside its capability scope receives a structured error rather than silent failure.

Richer MCP tool surfaces — federation operations, capability authoring via MCP, real-time subscription to atom updates, additional tools the small community asks for — are Phase 2.

## Alternatives Considered

### Alternative 1: Keep MCP server deferred to Phase 2 (the ADR-010 position)

- **Description:** MVP ships the substrate's local API and the `ffs` CLI as developer-audience surfaces. MCP server is Phase 2.
- **Pros:** Smaller MVP scope. Sharper focus on small-community validation. MCP server tool design can be informed by MVP usage.
- **Cons:** Home-claw absorption scenario is half-validated in MVP — agents write but don't read. Developers evaluating FFS see a substrate without the standardized agent-integration protocol the rest of the 2026 ecosystem uses.
- **Why rejected:** The home-claw absorption scenario's full validation requires bidirectional MCP. ADR-010 prioritized adoption-pacing over scenario-completeness; that's the wrong tradeoff for one of MVP's three motivating scenarios.

### Alternative 2: Ship a partial MCP server (e.g., read-only tools)

- **Description:** MVP ships `ffs_query`, `ffs_render_projection`, and `ffs_resolve_url` (the read tools). Defer `ffs_author_atom` and the audit/inspect tools to Phase 2.
- **Pros:** Smaller surface than the full six tools. Read-only is simpler.
- **Cons:** Half-shipping the MCP server creates an awkward middle. Agents that read substrate state may also need to write; cutting `ffs_author_atom` forces them back to filesystem-writes-to-ingest, which works but inconsistently with how they read. Better to ship the coherent six-tool set or keep the deferral.
- **Why rejected:** Coherence over partial delivery. The six tools form a tight set; cutting any of them creates an asymmetry that isn't a meaningful scope reduction.

## Consequences

### Positive

- The home-claw absorption scenario is fully validated in MVP. Agents read substrate state via MCP; agents write to ingest via filesystem; the substrate handles both paths correctly.
- Developers evaluating FFS see the standardized 2026 agent-integration protocol working, not just the CLI.
- The both-audiences-first-class commitment (ADR-002) is honored substantively rather than nominally.
- Phase 2 adds richer MCP tools informed by MVP usage rather than starting MCP integration from scratch in Phase 2.

### Negative

- MVP scope grows. The six MCP tools with capability checks are real implementation work, even as a thin wrapper.
- The MCP server is one more surface to maintain — bug fixes, capability-check correctness, error-message clarity.

### Risks

- The MCP server's tool surface turns out to be wrong for what agents actually need. **Mitigation:** Phase 2 explicitly reserves capacity for richer MCP tools informed by MVP usage. The six tools are the conservative starter set; richer ones come from real demand.
- Capability-check enforcement at the MCP boundary has subtle bugs that allow unauthorized authoring. **Mitigation:** Capability checks are evaluated by the same fixed evaluator that the substrate's local API uses; the MCP server is a thin wrapper that delegates capability evaluation, not a separate enforcement layer.

## Implementation Notes

The MCP server runs alongside the local daemon. Agents connect via the standard MCP transport (stdio or SSE per the agent's configuration). The Obsidian plugin and CLI continue to use the substrate's local API directly; the MCP server is for MCP-aware external agents.

Capability checks delegate to the substrate's existing capability evaluator. The MCP server constructs the same query that the local API would receive, applies the agent's identity, and returns capability-filtered results. There is no parallel capability machinery in the MCP layer.

The six MVP tools are the conservative starter set. Phase 2 adds tools as MVP usage reveals what's needed.
