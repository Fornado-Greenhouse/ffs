# ADR-009: Claw integration via OpenClaw or Hermes pattern

## Status

Accepted

## Date

2026-05-05

## Context

FFS's agents — the scribe, the librarian, the auditor, and the courier (Phase 2) — need a host process. The host runs them, manages their identity, mediates their access to LLMs, routes their interactions across user-facing channels (chat, email, voice), and handles operational concerns (lifecycle, resource limits, isolation).

Through 2025-2026, an industry pattern emerged for hosting AI agents on personal hardware: the **claw** pattern. A long-running daemon, multi-channel routing, persistent agent identity and memory, skills as `SKILL.md` artifacts, workspace isolation per agent, bring-your-own-LLM, privacy-default with data on the user's hardware.

OpenClaw (Peter Steinberger, 2025-2026) is the canonical claw implementation, with substantial GitHub adoption. Hermes Agent (Nous Research) is a sibling implementation with a built-in self-improvement loop. NemoClaw (NVIDIA) wraps OpenClaw with policy-based privacy guardrails.

FFS has a choice: build its own agent host, or contribute agent definitions to an existing claw.

## Decision

FFS does not reimplement the claw. FFS contributes **agent definitions** packaged as `SKILL.md`-shaped artifacts that run inside an existing claw (OpenClaw or Hermes). The agent definitions are FFS-specific in that they read from and write to FFS atom storage with FFS semantics; they are claw-pattern compatible in that they're packaged in a format the host can load.

The MVP implementation choice between an OpenClaw integration and a minimal FFS-specific daemon is operational, not architectural. Both are supported by the architecture; the decision is made at implementation time based on the relative maturity of OpenClaw's skill system and the cost of a parallel implementation.

## Alternatives Considered

### Alternative 1: Build FFS's own agent host from scratch

- **Description:** Implement a dedicated FFS daemon that hosts the FFS agents, handles identity and memory, mediates LLM access, etc.
- **Pros:** Full design control. Tight integration with FFS-specific concerns.
- **Cons:** Substantial implementation effort to replicate what OpenClaw and Hermes already do. Doesn't benefit from the claw ecosystem (multi-channel routing, existing skill libraries, user familiarity).
- **Why rejected:** Reinventing existing infrastructure. The claw pattern is canonical; FFS should be a citizen of it, not a competitor.

### Alternative 2: Build a minimal FFS-specific daemon as the only option

- **Description:** Ship only the minimal-FFS-daemon implementation. Don't support OpenClaw integration.
- **Pros:** Sharper FFS focus. Less dependency on external claw projects.
- **Cons:** Cuts off users who already run OpenClaw and want FFS to run inside it. Doubles the effort to maintain both a daemon and skills versus letting OpenClaw host the skills directly.
- **Why rejected:** The choice is operational. Both paths should be supported; the implementation decides based on practical considerations at build time.

## Consequences

### Positive

- FFS leverages the claw ecosystem (OpenClaw, Hermes, NemoClaw) rather than competing with it.
- Users who already run a claw can install FFS skills into their existing workspace without a separate daemon.
- Multi-channel routing (WhatsApp, Telegram, Discord, voice) comes for free via the claw layer.
- The FFS agents are portable across claw implementations because they conform to the `SKILL.md` pattern.

### Negative

- FFS's agent capabilities are constrained by what the chosen claw supports. Custom needs (e.g., FFS-specific resource limits) must be expressed in claw-compatible terms.
- Two integration paths (OpenClaw and minimal-FFS-daemon) means two installation experiences to document and support.

### Risks

- OpenClaw or Hermes pivots in incompatible ways. **Mitigation:** Both are open-source under active community governance. The minimal-FFS-daemon path provides an escape if the claw landscape destabilizes.
- The `SKILL.md` format diverges across claws. **Mitigation:** FFS skills target a common subset; claw-specific extensions are isolated.

## Implementation Notes

An FFS agent definition is a directory containing:

```
ffs-scribe/
├── SKILL.md              # metadata: type, capabilities, schedule, entry points
├── definition.atom.json  # the FFS agent definition atom (signed, versioned)
├── extraction.py         # the actual scribe logic
├── prompts/              # LLM prompts, edition-able
└── tests/                # behavioral tests for this agent's outputs
```

The `SKILL.md` follows OpenClaw's conventions. The `definition.atom.json` is FFS's typed contract — predicate scope, entity scope, capability grants, resource limits, signing key reference. Both files are different views of the same thing.

The MVP daemon implementation choice (OpenClaw or minimal FFS) does not affect the agent definitions; the same skills load into either host.

This ADR replaces the standalone `CLAW.md` document from prior architecture iterations. The claw integration is a single architectural decision, properly scoped here.
