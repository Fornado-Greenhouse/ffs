# ADR-002: Both audiences first-class

## Status

Accepted

## Date

2026-05-05

## Context

FFS serves two audiences with different needs: application developers building products like Folodex on top of the substrate, and end users running FFS directly on their own hardware. Prior architecture iterations were ambivalent about which audience was primary, with the document quietly drifting toward whichever audience had been most recently discussed.

The PRD must commit. The choice has real consequences: a developer-first substrate emphasizes APIs, SDKs, MCP exposure, and minimal end-user surface; an end-user-first substrate emphasizes the Obsidian plugin, opinionated path libraries, and CLI ergonomics; a both-first substrate must ship both at comparable polish, which is more work but produces a more credible substrate.

## Decision

Both audiences are first-class. The MVP must deliver functional surfaces for both: a working CLI and `ffs://` URL scheme for developers, and a working Obsidian plugin with the projection surface for end users. Neither audience is deferred.

## Alternatives Considered

### Alternative 1: Developers first, end users come through their apps

- **Description:** Ship a substrate with strong API surface and minimal end-user tooling. Folodex and other applications provide the user-facing experience; FFS-the-substrate is invisible to them.
- **Pros:** Smaller MVP. Sharper developer-audience focus. Apps can take responsibility for their UX without the substrate prescribing it.
- **Cons:** Developer-only substrates rarely get adopted because there's nothing for an interested developer to *try* directly. The end-user surfaces (Obsidian plugin, CLI, projection navigation) are how a developer evaluates whether the substrate's claims hold up. Without them, the substrate is abstract.
- **Why rejected:** The technically-capable end user is also a primary persona; cutting their surface cuts off self-evaluation by developers and self-onboarding by power users.

### Alternative 2: End users first, developers come later

- **Description:** Ship a polished end-user product (Obsidian plugin, projection surface, federation UX). Add developer surface in Phase 2.
- **Pros:** Sharper end-user proposition. Easier to demonstrate to non-technical peers.
- **Cons:** Developers building the next applications on FFS need a stable substrate API and addressing scheme to start their work. Deferring those means the second-app and third-app experiences don't get tested in MVP.
- **Why rejected:** The success criterion (small community using federation) implicitly requires that applications like Folodex exist, which means developers need the substrate.

## Consequences

### Positive

- Developers evaluating FFS can install it, run the CLI, navigate the projection surface, and verify the architecture's claims directly.
- End users (the technical half of the mixed-capability community) get a working day-one experience without waiting for applications to mature.
- The substrate's API and end-user surface co-evolve, which forces them to be coherent rather than divergent.

### Negative

- Larger MVP scope than either audience-first approach.
- Both surfaces must be at comparable polish, which means neither can be skimped on.
- The Obsidian plugin specifically is load-bearing for the end-user surface, which makes it a critical-path item.

### Risks

- Splitting attention across two audiences could result in mediocre delivery to both. **Mitigation:** The substrate-first approach (ADR-003) means both surfaces are functional but not polished — neither audience gets the deep-investment treatment in MVP, which keeps scope tractable.

## Implementation Notes

The CLI and Obsidian plugin are co-equal MVP deliverables. The MCP server, which is the most leveraged developer-audience surface, is explicitly Phase 2 (see ADR-010) — meaning developers in MVP work through the local API and CLI, not MCP. This is a pragmatic compromise that keeps MVP scope sane while honoring both-audiences-first-class.
