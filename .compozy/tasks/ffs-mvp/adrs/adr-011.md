# ADR-011: Path library starts at three (contacts/people/notes)

## Status

Accepted

## Date

2026-05-05

## Context

The opinionated path library is what makes the substrate legible as a navigable place rather than a database. Each path family — `contacts/`, `decisions/`, `projects/`, `people/`, `questions/`, `action-items/`, `policies/` — has a defined query, rendering convention, pagination strategy, and frontmatter convention.

Prior architecture iterations committed to a starter library of seven path families. The YAGNI question is whether all seven are needed in MVP, given the success criterion is contact-graph sovereignty with a small community.

## Decision

The MVP path library ships three families:

- **`contacts/`** — for the contact-graph use case. Sub-paths: `recent/`, `starred/`, `by-org/`, `by-name/`, `phone/`, `email/`, `from/<peer>/` (federation), `intersection/with/<peer>/`, `all/`.
- **`people/`** — for general person-entity references that aren't contacts (colleagues mentioned in notes, etc.). Sub-paths: `recent-collaborators/`, `by-team/`, `by-org/`, `all/`.
- **`notes/`** — a generic catch-all for content that doesn't fit `contacts/` or `people/`. The path absorbs scribe-promoted notes from the ingest folder. Sub-paths: `recent/`, `by-tag/`, `all/`.

Decisions, projects, questions, action-items, and policies are Phase 2.

## Alternatives Considered

### Alternative 1: All seven path families in MVP

- **Description:** Ship the full starter library — contacts, decisions, projects, people, questions, action-items, policies — at MVP.
- **Pros:** Comprehensive starter coverage. Users with diverse use cases find paths that fit them.
- **Cons:** Each path family requires defining the query, the rendering convention, the pagination strategy, the frontmatter convention. Five additional path families is real work. The success criterion is contact-graph sovereignty; adding path families that don't serve that criterion is scope creep.
- **Why rejected:** YAGNI. The MVP path library should serve the success criterion plus a flexible catch-all; richer libraries belong in Phase 2 informed by what the community actually needs.

### Alternative 2: Only contacts and people

- **Description:** Ship only the relationship-graph paths. No catch-all.
- **Pros:** Sharpest contact-graph focus.
- **Cons:** Without `notes/`, users have nowhere to put content that doesn't fit. They're either blocked from capture or forced to use `contacts/` and `people/` paths inappropriately.
- **Why rejected:** The flexible catch-all is necessary. Without it, MVP's home-claw-absorption scenario (agents writing markdown that may be about anything) has nowhere to land cleanly.

### Alternative 3: Just a generic path library extension mechanism with one example

- **Description:** Ship `contacts/` only as a worked example; emphasize the path library is forkable and users can author their own path-definition atoms for whatever they need.
- **Pros:** Strongest signal that the library is user-extensible.
- **Cons:** Most users won't author path-definition atoms in MVP. Without `notes/`, capture has nowhere convenient to land. The forkable-from-day-one framing is right; reducing the starter library to a single example overshoots.
- **Why rejected:** Three is small enough to be honest about MVP scope while large enough to handle the actual use cases.

## Consequences

### Positive

- MVP path library is sized to the success criterion. Contacts and people serve the contact-graph use case directly; notes serves agent-absorption and general capture.
- Phase 2 path additions can be informed by what the small community actually requests. We don't pre-commit to specifications that may turn out wrong.
- The forkable-from-day-one property is real: users can author path-definition atoms for new paths whenever they want, without waiting for FFS Phase 2.

### Negative

- Users who want decision-tracking, project-tracking, action-item-tracking, etc., must either fork their own paths or wait for Phase 2.
- The MVP feels less "complete" than a full starter library would.

### Risks

- The community's actual needs in Phase 2 don't match the deferred path families (decisions, projects, questions, action-items, policies). **Mitigation:** This is exactly the reason to defer them. Phase 2 priorities emerge from real use, not from up-front guessing.

## Implementation Notes

Path-definition atoms specify the query, rendering convention, pagination strategy, and frontmatter convention for each path. The MVP ships path-definition atoms for the three families above as part of the starter library, authored by FFS itself and superseded only by user action.

User-authored path definitions are a power-user feature in MVP — possible but not exposed in any UI beyond CLI. Phase 2 may add a richer authoring experience.
