# ADR-001: Records-shaped substrate, not file-shaped

## Status

Accepted

## Date

2026-05-05

## Context

The substrate's fundamental data model has two viable shapes: records (small, structured units of information with attribution and time, the iCloud Notes / Apple Contacts / Datomic lineage) or files (larger, unstructured content with directories and modification dates, the OneDrive / Dropbox / Box / Google Drive / iCloud Drive lineage).

The choice is foundational. It determines how access control works, how concurrent writes resolve, how history is preserved, and how applications above the substrate compose. It's not a styling choice; it's an architectural identity.

The OneDrive-lineage products spent the past two decades fighting save-pattern complexity, merge conflicts, audit gaps, and access-control granularity problems. Their failures are well-documented. The Notes-lineage products quietly succeeded at sub-document granularity, concurrent multi-author writing without merge conflicts, and field-level history.

## Decision

FFS is records-shaped. The substrate stores **atoms** — signed, time-stamped, classified records about entities. Files appear in FFS only as surfaces (ingestion folder, projection folder, configuration folder), never as the substrate itself.

## Alternatives Considered

### Alternative 1: Files-shaped substrate

- **Description:** Store knowledge as markdown files with frontmatter, the way Obsidian and similar tools do. Add metadata, history, and access control on top via plugins or sidecar systems.
- **Pros:** Familiar mental model. Tools work without special integration. Easy to bootstrap with existing markdown editors.
- **Cons:** Re-creates the OneDrive failure mode. Sub-document classification, concurrent writes, fine-grained capability are all hard or impossible without abandoning the file model. The save-pattern complexity (Word's atomic-save dance, etc.) becomes the substrate's problem.
- **Why rejected:** Two decades of failure across the OneDrive lineage indicate this path is exhausted. The architectural needs that motivate FFS specifically require sub-record granularity.

### Alternative 2: Hybrid

- **Description:** Store records but expose the file as the primary unit at the user surface; pretend records are files when convenient.
- **Pros:** Could ease user mental-model transition.
- **Cons:** The hybrid is conceptually unclear and operationally complex. Either the records win (and the file metaphor leaks awkwardly) or the files win (and we have the OneDrive problem).
- **Why rejected:** Conceptual clarity matters more than mental-model familiarity. The three folder spaces (ingestion, projection, configuration) provide enough file-shaped surface to handle user expectations without conflating substrate with surface.

## Consequences

### Positive

- Concurrent writers (humans, AI agents, federation peers) produce atoms without merge conflicts; supersession is explicit.
- Sub-record classification enables tier-based selective sharing — the contact-graph use case becomes tractable.
- Bitemporal history is structural rather than bolted on.
- Capability-as-data composes cleanly because authorization targets atoms, not files.

### Negative

- Users must understand the records-with-projections model, which is unfamiliar. The Obsidian plugin and projection surface are partly UX work to manage this.
- Tools that expect files (text editors, search indexers) interact via materialized projections rather than directly with the substrate. The daemon manages this, but it's complexity that wouldn't exist with a files-shaped substrate.

### Risks

- Users may be confused by the records-but-files-on-disk dual model. **Mitigation:** Visual treatment in the Obsidian plugin distinguishes projections from notebook entries; the three folder spaces have non-overlapping semantics enforced by structure.

## Implementation Notes

The substrate's atom format includes entity reference, predicate, claim content, author identity, signature, valid time, transaction time, classification, provenance, and supersession links. Content addressing uses the multihash format for forward compatibility with future hash algorithms.
