# ADR-006: `ffs://` URL scheme as public stable contract

## Status

Accepted

## Date

2026-05-05

## Context

The substrate needs a way for tools, scripts, agents, and humans to address atoms and projections from outside the substrate. Three options:

- **No URL scheme.** Tools work through SDK calls or local API. References between content are SDK-specific. Each tool reinvents addressing.
- **Internal addressing only.** A scheme exists but is not stable or documented. Subject to change. Tools depend on FFS internals.
- **Public stable URL scheme.** A documented, stable scheme that any tool can reference, modeled on `s3://`'s journey from AWS-internal to industry-universal addressing.

`s3://` is the relevant precedent. AWS could have shipped only the SDK; the URL scheme became infrastructure because tools could refer to S3 objects in a uniform way. The scheme outlived any particular implementation.

## Decision

FFS commits to `ffs://` as a public, stable contract. The form:

```
ffs://<graph>/<address>[?<query>]
```

Three addressing modes:

- **Path** addressing: `ffs://my-graph/contacts/by-name/S/`
- **Atom** addressing: `ffs://my-graph/atom/<content-hash>`
- **Entity** addressing: `ffs://my-graph/entity/<entity-id>`

Bitemporal queries are URL parameters: `?as_of=2026-04-15`, `?valid_at=2026-Q1`.

The scheme is a noun, not a verb. It addresses things; it does not perform operations. Tools apply verbs (cat, ls, get) to URLs.

The MVP ships an `ffs` CLI that resolves `ffs://` URLs as a single static binary per platform.

## Alternatives Considered

### Alternative 1: No URL scheme

- **Description:** Tools work through SDK calls or local API. No universal URL form.
- **Pros:** Less commitment. Less to keep stable.
- **Cons:** Cuts off shell-pipeline use, cross-tool references, addressing in chat/email/docs, the Linux community's natural addressing idiom. Tools that want to reference FFS content must reach for FFS-specific APIs.
- **Why rejected:** The Linux community vision and the Windows simpleton case both depend on a universal addressing form. Without `ffs://`, the substrate is not fully a file system in the way it claims to be.

### Alternative 2: Versioned scheme (`ffs1://`, `ffs2://`)

- **Description:** Version the scheme prefix to allow breaking changes.
- **Pros:** Explicit versioning. Easier to evolve.
- **Cons:** Versioning the prefix signals impermanence. Users typing `ffs1://` once and `ffs2://` later is a worse experience than typing `ffs://` indefinitely. The bitemporal query parameters and addressing modes have enough flexibility to evolve within one stable prefix.
- **Why rejected:** Stability is the strategic value. Versioning the prefix undermines the "modeled on `s3://`" framing.

## Consequences

### Positive

- Tools that have nothing to do with FFS can resolve `ffs://` URLs given the CLI binary. Adoption requires only dropping the binary in `$PATH`.
- Shell pipelines work: `ffs ls ffs://my-graph/contacts/recent/ | xargs ffs cat`.
- Linux community has its addressing idiom; Windows users have a stable handle that survives reorganization of underlying storage.
- Cross-tool references work: an `ffs://` URL pasted into a chat or email is meaningful and resolvable.
- Bitemporal queries are typeable: `?as_of=2026-04-15` is something a user can add by hand without SDK gymnastics.

### Negative

- The scheme is a public commitment. Breaking changes are off the table. Backward compatibility is a permanent constraint.
- Defining the addressing modes precisely up front is hard; cases not anticipated may force awkward extensions later.

### Risks

- We commit to a scheme prematurely and discover a fundamental flaw. **Mitigation:** The three addressing modes (path, atom, entity) plus bitemporal query parameters cover the cases the architecture has identified across all motivating scenarios. The risk is small.
- Tools other than the FFS CLI may resolve `ffs://` URLs incorrectly. **Mitigation:** The CLI is the reference resolver in MVP; documentation specifies the resolution semantics for third-party tools that want to implement them.

## Implementation Notes

The `ffs` CLI is a separate binary from the substrate daemon. It speaks to the local daemon (or remote substrates over HTTPS) to resolve URLs. The CLI is shell-friendly: pipes, exit codes, plain output by default, `--json` for structured output.

Federation URLs (`ffs://peer-graph/...` for graphs other than the local one) resolve through the same machinery; the local daemon proxies queries to the appropriate peer per bridge contracts.
