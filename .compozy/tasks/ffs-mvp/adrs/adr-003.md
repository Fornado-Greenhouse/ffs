# ADR-003: Substrate-First MVP

## Status

Accepted

## Date

2026-05-05

## Context

Three MVP approaches were considered for FFS:

- **Substrate-First MVP:** ship the substrate with working surfaces for each audience; functional but not polished; validate end-to-end via real use; learn from a small community.
- **Polish-First MVP:** ship fewer features at higher polish; defer some scenarios to focus on making one slice excellent.
- **Community-First MVP:** design straight for the 5-20 person small-community workflow; ship multi-peer federation aggregation in MVP; accept that individual-user polish is Phase 2.

The choice determines what the small community will actually receive in MVP, and what gets deferred to Phase 2.

## Decision

FFS adopts the Substrate-First MVP approach. The MVP ships:

- The substrate's core primitives (atoms, capability, classification, signing, supersession, erasure, federation).
- Working surfaces for both audiences (Obsidian plugin functional but minimal; CLI fully functional with `ffs://` URL resolution).
- Federation between two peers bilaterally with the five MVP federation features (capability authoring, peer mounting, tier-based selective sharing, intersection computation, revocation).
- The three motivating scenarios validated end-to-end: contact-graph sovereignty, home-claw absorption, day-one-clone-and-collaborate.

Polish is deferred. Multi-peer federation aggregation is deferred. The MCP server is deferred. Direct projection editing is deferred.

## Alternatives Considered

### Alternative 1: Polish-First MVP

- **Description:** Ship fewer features, each one polished. Obsidian plugin with rich navigation and edit absorption; CLI comprehensive; federation between two peers polished to delight. Defer some motivating scenarios (probably home-claw absorption) to Phase 2.
- **Pros:** Each surface feels finished. Easier to demo. Sharper user impression.
- **Cons:** Cutting a motivating scenario weakens the substrate's identity (all three scenarios share architecture; cutting one removes a use case the substrate already supports). Polish without breadth means the substrate's distinctive claims remain unproven in the use cases that don't ship.
- **Why rejected:** The success criterion is small-community validation across the substrate's commitments, not a polished slice. Polish-first risks demoing well but proving little.

### Alternative 2: Community-First MVP

- **Description:** Design straight for the small-community workflow. Ship multi-peer federation aggregation in MVP, fully fleshed-out contact-graph use case, both surfaces unpolished.
- **Pros:** Most directly targets the success criterion. Federation is the sharpest distinctive feature; emphasizing it in MVP makes sense.
- **Cons:** Multi-peer aggregation is meaningful complexity that adds risk to MVP. Bilateral federation already validates the architecture's federation claims; multi-peer is UX leverage on top of the working primitive. Building the leverage before validating the primitive is premature.
- **Why rejected:** Scope discipline. MVP should validate the primitive before investing in features that depend on the primitive working.

## Consequences

### Positive

- MVP scope is sized to validate the architecture without over-investing in polish that may turn out to be wrong.
- The 5-20 person community gets something real to use; their feedback shapes Phase 2.
- All three motivating scenarios are exercised in MVP, providing breadth coverage of the substrate's commitments.
- Phase 2 priorities emerge from real use rather than from up-front guessing.

### Negative

- Surfaces feel rougher than they would under Polish-First. Demo experience is less polished.
- Multi-peer federation aggregation, the most directly community-oriented federation feature, is Phase 2 — community workflows in MVP are pair-by-pair.

### Risks

- Functional-but-not-polished surfaces could leave a bad first impression that prevents adoption. **Mitigation:** The technical-friend onboarding model (ADR-002 implication) means each new user is supported through initial roughness; rough edges become onboarding feedback rather than abandonment.

## Implementation Notes

"Functional but not polished" is the operating principle for each MVP surface. The Obsidian plugin must work (folder enumeration, projection rendering, ingest absorption, daily health summary) but doesn't need rich substrate search, faceted search, real-time invalidation, or working-set heuristics. The CLI must work for the documented operations but doesn't need extensive subcommands. Federation must work for the five MVP features but doesn't need multi-peer aggregation.
