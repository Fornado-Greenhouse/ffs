# ADR-025: macOS .app bundle wrapping for restricted-entitlement CLIs

## Status

Accepted

## Date

2026-06-19

## Context

ADR-023 committed FFS to a Developer-ID-signed binary + `keychain-access-groups` entitlement so the launchd-spawned daemon and the interactive CLI share one keychain bucket. The implementation it prescribed — embed the provisioning profile into the Mach-O via `-Wl,-sectcreate,__TEXT,__provisioning,…` and let AMFI read it at exec — is **folklore.** AMFI does not look in that section. The recipe survived on forum posts and in tooling like Twocanoes-derived guides because the binaries those guides were used with were also placed inside `.app` bundles that already had the profile as a file, so AMFI found it via the file path and the section was irrelevant.

The empirical sequence on 2026-06-19, macOS 26.2, signing identity `Developer ID Application: Alexander Foley (3S9R9K2L38)`:

| Configuration | Outcome |
|---|---|
| Signed Developer ID, hardened runtime, notarized, no entitlements | Daemon launches normally, emits expected stderr |
| Same binary + `keychain-access-groups` entitlement claim, no profile | `SIGKILL` (exit 137, zero stderr) — expected, AMFI rejects restricted claim |
| Same binary + entitlement + profile embedded via `__TEXT,__provisioning` Mach-O section | **`SIGKILL`, identical outcome** — AMFI ignored the section |
| Plus `__TEXT,__info_plist` section embed (CFBundleIdentifier=com.fornado.ffs) | `SIGKILL` still |
| Plus matching the binary's `application-identifier` + `team-identifier` entitlements to the profile's exactly | `SIGKILL` still |
| Re-issuing the profile to bind the LOCAL signing cert's SHA-1 (`7E65…` instead of the other Developer ID Application cert in the portal) | `SIGKILL` still |
| Dropping `app-sandbox = false`, claiming `keychain-access-groups = 3S9R9K2L38.*` (the literal wildcard the profile authorizes) | `SIGKILL` still |
| Removing entitlements entirely | Daemon launches |

The behavior is: AMFI rejects restricted-entitlement claims for any raw Mach-O on macOS regardless of what's embedded into the binary's sections.

Quinn "Eskimo" of Apple DTS, [thread/743979](https://developer.apple.com/forums/thread/743979) (paraphrased — *standalone executables cannot use restricted entitlements; embed in app-like wrapper*), and the canonical doc *[Signing a daemon with a restricted entitlement](https://developer.apple.com/documentation/xcode/signing-a-daemon-with-a-restricted-entitlement)*. The provisioning profile has to be a file at `Contents/embedded.provisionprofile` inside a `.app` bundle. There is no other path Apple supports.

## Decision

Wrap the three FFS macOS binaries in a single `.app` bundle:

```
FFS.app/
  Contents/
    Info.plist                    # CFBundleIdentifier=com.fornado.ffs, LSUIElement=true
    embedded.provisionprofile     # the Apple-issued Developer ID profile, AS A FILE
    MacOS/
      ffs                         # CFBundleExecutable — primary CLI
      ffs-daemon                  # additional Mach-O in the same bundle
      ffs-mcp                     # additional Mach-O in the same bundle
    _CodeSignature/               # produced by `codesign --force --deep`
```

This is Apple's "daemon in app's clothing" pattern. `LSUIElement=true` keeps it from registering as a dockable GUI app. All three binaries share the bundle's App ID + provisioning profile; codesigning the bundle once recursively signs every Mach-O under `Contents/`.

Symlinks from the user's `$PATH` (`~/.local/bin/ffs` → `…/FFS.app/Contents/MacOS/ffs`, same for the other two) preserve the CLI ergonomics. The installer creates these symlinks at install time; uninstall removes them.

## Alternatives Considered

### Alternative 1 — keep ADR-023's section-embed path, just document the kill is OK

- **Description:** Accept that the macOS keychain path doesn't actually work in MVP, document the SIGKILL as known, fall through to Path A (env-var keys + file-based fallback) on macOS.
- **Pros:** Zero further code work. The signing + notarization infrastructure built for task_33 still delivers value (signed/trusted releases for Gatekeeper).
- **Cons:** Defeats the entire point of ADR-023. The launchd identity-drift bug task_27 was scoped to solve resurfaces. Documentation has to pretend the broken section embed is "deferred" rather than "incorrect."
- **Why rejected:** User explicitly chose to chase the canonical Apple path tonight. Also: this would leave a bug in the code (section-embed build.rs files emit linker args that do nothing) and a wrong ADR-023.

### Alternative 2 — privileged helper via SMAppService / SMJobBless

- **Description:** Ship a minimal helper bundle that runs as root and brokers keychain access for the CLI via XPC. Pattern that 1Password CLI uses.
- **Pros:** Cleaner privilege separation. Works without bundle-wrapping the user-facing CLIs.
- **Cons:** Substantially more architectural work (XPC service, helper installation flow, launchd helper management, privilege check on every keychain operation). The CLI then has to make IPC calls rather than direct calls. The simple "two helpers in `ffs-core::store::keyring_macos`" surface area becomes "client + server + XPC plumbing."
- **Why rejected:** Out of MVP scope. Bundle wrapping is the simpler Apple-blessed path for this need.

### Alternative 3 — file-based shared secret on macOS only (Alternative B from chat)

- **Description:** Skip the keychain integration on macOS. Store keys at `~/.ffs/secrets/{owner_key,dek}` mode 0600, the way Path A in the technical-friend-checklist describes.
- **Pros:** Zero macOS-specific code. Same shape as the Linux+Windows fallback paths.
- **Cons:** Weaker security posture — files are inside `$FFS_DATA_DIR` which the user is likely to back up or sync. The whole point of the keychain integration was to keep secrets outside the backup boundary.
- **Why rejected:** Same as ADR-023 originally rejected the file-only approach. Defense-in-depth on the secrets matters here.

## Consequences

### Positive

- The keychain path actually works after this lands. Daemon + CLI + MCP all see the same access group; the launchd identity-drift bug task_27 was scoped against is fixed for real.
- Apple-blessed mechanism. Won't drift under future macOS releases the way folklore-section approaches do.
- The bundle is the natural artifact Apple's `notarytool` + `stapler` were designed for. `stapler staple FFS.app` actually attaches a ticket to the bundle (raw Mach-O can't have one stapled), so first-launch on a fresh machine doesn't need network to Apple's CDN.

### Negative

- `target/release/` artifact layout changes. Build scripts, installer, launchd plist, CI workflow all touch the new path.
- `~/.local/bin/ffs` becomes a symlink rather than the actual binary. Cosmetic but worth flagging in install docs.
- Each binary's `codesign --display` output will show the bundle's Info.plist rather than per-binary identifiers. That's the intended pattern but worth knowing for diagnostics.

### Risks

- **Bundle path under launchd.** The `com.ffs.daemon.plist` `ProgramArguments` array has to point at `…/FFS.app/Contents/MacOS/ffs-daemon` — a path inside the user's home if the bundle installs to `~/.local/Applications/`. Test that launchd is willing to run that path (it should be; this is how every user-installed Mac app works).
- **Codesign deep mode interactions.** `codesign --force --deep --options runtime --entitlements …` on a bundle has subtle semantics — the bundle's Info.plist is the source of truth for CFBundleIdentifier even if individual Mach-O sections also embed one. Code review the script against the official doc to make sure we're not double-stamping.
- **CI cache invalidation.** The release.yml workflow's `Swatinem/rust-cache@v2` step keys off `target/`. Bundle creation writes outside `target/release/<binary>` so we don't lose cache hits, but the new staging path needs a thought-through `.gitignore` so we don't accidentally commit `FFS.app/`.

## Implementation Notes

`Info.plist` minimum keys (`LSUIElement` is the load-bearing one for a headless CLI bundle):

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>CFBundleIdentifier</key>           <string>com.fornado.ffs</string>
    <key>CFBundleName</key>                 <string>FFS</string>
    <key>CFBundleExecutable</key>           <string>ffs</string>
    <key>CFBundlePackageType</key>          <string>APPL</string>
    <key>CFBundleShortVersionString</key>   <string>0.1.0</string>
    <key>CFBundleVersion</key>              <string>1</string>
    <key>LSUIElement</key>                  <true/>
    <key>LSMinimumSystemVersion</key>       <string>11.0</string>
</dict>
</plist>
```

`scripts/codesign-macos.sh` will be rewritten to (1) construct the bundle from the build's output, (2) place `embedded.provisionprofile` as a file at `Contents/`, (3) place `Info.plist` at `Contents/`, (4) `codesign --force --deep --options runtime --timestamp --entitlements … --sign … FFS.app`. The current per-Mach-O loop goes away.

`crates/{ffs-daemon,ffs-cli,ffs-mcp}/build.rs` files get deleted — they emit linker args that AMFI doesn't actually read, and the entire section-embed approach was wrong.

Notarization shape changes from per-binary `.zip`s to a single `FFS.app.zip` submission. After acceptance, `xcrun stapler staple FFS.app` attaches the ticket directly (this works for bundles where it didn't work for raw Mach-O).

ADR-023 stays in the tree as the **intent** document — its decision to use shared access groups + codesigning is upheld. Its implementation specifics (Mach-O section embed) are superseded by this ADR. ADR-023's references section gains a "Superseded for implementation by ADR-025" pointer.
