# ADR-021: Predicate spec format — TOML with embedded JSON Schema

## Status

Accepted

## Date

2026-05-05

## Context

Predicate specs define the substrate's vocabulary. Each spec carries:

- **Identity**: a stable predicate name (`contact.person.work_email`).
- **Claim schema**: validation rules for the structured payload of atoms claiming this predicate.
- **Rendering convention**: how atoms claiming this predicate are rendered into projection markdown (frontmatter, body sections, list patterns).
- **Reverse-map annotations**: which rendered output element maps back to which atom field (the load-bearing input to the fast-path edit classifier per ADR-014).
- **Pagination strategy**: for predicates that drive projection paths (contacts, people, notes), how the path enumerates entities (alphabetical, recency, organization).
- **Frontmatter convention**: which atom fields surface as YAML frontmatter on rendered projections.

Predicate specs live in the configuration folder (`~/.ffs/config/predicates/`), are git-versioned, and are edited by humans. They are also superseded the same way other atoms are: a new spec atom replaces an old one, both remain auditable.

## Decision

Predicate specs are TOML files, one per predicate, located at `~/.ffs/config/predicates/<predicate_name>.toml`. The TOML structure carries identity, rendering, reverse-map, pagination, and frontmatter metadata. Claim payload validation uses JSON Schema (Draft 2020-12) embedded in a `[claim_schema]` section as a TOML table that round-trips to JSON.

Example skeleton:

```toml
name = "contact.person"
version = 1
parent_predicate = "person.generic"

[claim_schema]
"$schema" = "https://json-schema.org/draft/2020-12/schema"
type = "object"
required = ["display_name"]
[claim_schema.properties]
display_name = { type = "string" }
work_email = { type = "string", format = "email" }
phone = { type = "string" }
notes = { type = "array", items = { type = "string" } }

[rendering]
template = "contact-person.md.tera"
frontmatter_fields = ["display_name", "work_email", "tier"]

[[reverse_map]]
output = "frontmatter.display_name"
atom_field = "claim.display_name"
edit_kind = "single_line_text"

[[reverse_map]]
output = "section.notes.list_item"
atom_field = "claim.notes[]"
edit_kind = "additive_section"

[pagination]
strategy = "alphabetical_first_letter"
group_field = "claim.display_name"
```

Spec files are committed to git via the configuration folder. The daemon loads them at startup, watches for changes, and reloads when files are modified. A spec change is itself authored as a substrate atom (the predicate spec atom) so the change is bitemporal and signed.

The MVP starter library ships three predicate-spec files: `contact.person.toml`, `person.generic.toml`, `note.toml` (with sub-predicate variants where needed for tier-based classification of contact fields).

## Alternatives Considered

### Alternative 1: Single-format JSON files with JSON Schema inline

- **Description**: One JSON file per predicate. Same content as the TOML option but uniform with the atom envelope format (ADR-017).
- **Pros**: One format end-to-end. Aligns with canonical-JSON commitment. JSON Schema is native.
- **Cons**: JSON is awkward for hand editing — no comments, strict trailing-comma rules, escape soup for any string with quotes. Predicate specs are read and edited frequently; a friendlier surface format pays off. The atom-envelope format and the human-config format are different concerns; the substrate already has multiple formats (markdown projections, JSON envelopes, etc.).
- **Why rejected**: Hand editability matters for the config surface even if it does not for the atom envelope.

### Alternative 2: YAML files with embedded JSON Schema

- **Description**: YAML structure with JSON Schema for claim validation.
- **Pros**: Most human-friendly editing. Broadly familiar.
- **Cons**: YAML's parser ambiguity is a known footgun: `version: 1` parses as integer; `version: 1.0` parses as float; `country: NO` parses as `False` (Norway problem). Predicate specs are config that affects substrate semantics; YAML's surprises compound. TOML has explicit types and no surprises.
- **Why rejected**: TOML's strict typing makes it safer for config that gates substrate behavior.

### Alternative 3: Rust-defined predicates only (compiled-in)

- **Description**: Predicates are defined as Rust modules in `ffs-core`; users cannot add predicates without recompiling.
- **Pros**: Type-safe at compile time. No runtime spec parsing.
- **Cons**: Contradicts PRD § Core Features § Three folder spaces (configuration is git-versioned, edited by humans). Forks the substrate from its forkable-from-day-one property (ADR-011). Power users cannot define new predicates for their domain without modifying FFS source.
- **Why rejected**: Incompatible with PRD architecture.

## Consequences

### Positive

- Specs are hand-editable and git-friendly.
- TOML's strict typing avoids YAML's coercion bugs without sacrificing readability.
- JSON Schema for claim validation is well-tooled (`jsonschema` crate in Rust); validation is the same on the daemon side as it would be in any client that wants to validate before submitting.
- Reverse-map annotations are first-class structured data, not free-form comments — the fast-path classifier can rely on them.

### Negative

- Two formats inside the substrate (TOML for predicate specs, JSON for atom envelopes). Mitigated by clear scoping: TOML for human config, JSON for machine data.
- TOML tooling in browsers is weaker than YAML/JSON; visual spec editors (Phase 2 work) need to choose a tooling story.

### Risks

- A predicate-spec change breaks rendering or reverse-map invariants. **Mitigation**: spec atoms are bitemporal — old projections continue to render under the old spec until re-rendered; daemon validates new specs against a battery of golden-file tests before activating them.
- JSON Schema validators differ across implementations. **Mitigation**: pin the Rust validator (`jsonschema` crate) as the reference; conformance is the daemon's evaluation, not the client's.

## Implementation Notes

- The predicate-spec loader: `ffs-core::predicate::SpecLoader` reads TOML files from `~/.ffs/config/predicates/`, parses each, validates internal consistency (every reverse-map output references a defined rendering element; every JSON Schema is itself valid), and registers the spec.
- The rendering engine uses Tera templates referenced by the spec's `[rendering].template` field; templates live in `~/.ffs/config/templates/` (also git-versioned).
- Reverse-map rules drive the fast-path classifier: when a projection file changes, the daemon diffs against the last-rendered version, walks the reverse-map rules in order, and attempts to match the diff to one rule. A successful match produces a supersession atom; no match routes to ingest as a correction.
- Spec atoms (the on-substrate record of a predicate spec) carry the TOML content as their claim payload, hashed and signed like any other atom.
