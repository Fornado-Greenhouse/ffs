# ADR-017: Canonical JSON (RFC 8785 JCS) atom envelope

## Status

Accepted

## Date

2026-05-05

## Context

Atoms are signed, content-addressed records. The signature must cover a canonical byte sequence so that any verifier produces the same digest from the same logical atom. Content addressing requires the same canonical bytes for the multihash. The format also affects every layer that touches atoms: the SQLite store (envelope blob), federation transport, the MCP server's `ffs_query`, the CLI's `--json` output, and the predicate-spec validation pipeline.

ADR-001 establishes atoms as records. ADR-006 commits `ffs://` as a public stable contract; the atom format that backs `ffs://my-graph/atom/<hash>` is implicitly part of that contract because the hash and the format are coupled.

The format choice is a long-lived public commitment.

## Decision

The atom envelope is JSON, canonicalized per RFC 8785 (JSON Canonicalization Scheme, JCS). The signature covers the JCS-canonicalized bytes. The content address is `multihash(blake3(jcs_bytes))`. The envelope shape:

```json
{
  "v": 1,
  "entity": "<entity_id>",
  "predicate": "<predicate_name>",
  "claim": { ... structured payload validated against the predicate's claim_schema ... },
  "author": "<ed25519_public_key_multibase>",
  "valid_from": "2026-05-05T00:00:00Z",
  "valid_to": null,
  "tx_time": "2026-05-05T14:23:11.421Z",
  "classification": "<tier>",
  "supersedes": "<prior_atom_multihash | null>",
  "provenance": [{"kind": "ingest_file", "uri": "...", "hash": "..."}],
  "signature": "<ed25519_signature_multibase>"
}
```

Signing protocol:

1. Construct the envelope with `signature` field omitted.
2. JCS-canonicalize.
3. Sign the canonical bytes with the author's Ed25519 key.
4. Re-insert `signature` into the envelope.
5. JCS-canonicalize the full envelope (including `signature`).
6. Compute `blake3(jcs_bytes)` and wrap as multihash.

Verification reverses steps 1-3.

The hash that backs `ffs://<graph>/atom/<hash>` is the multihash from step 6.

## Alternatives Considered

### Alternative 1: Deterministic CBOR (RFC 8949 Section 4.2)

- **Description**: Encode atoms as CBOR with deterministic encoding rules; sign the bytes; multihash the bytes.
- **Pros**: ~30% smaller payloads. Faster parsing. Well-specified deterministic encoding.
- **Cons**: Binary format harder to debug. Tooling less universal — `jq` over CBOR requires conversion. The substrate's "human-inspectable" character (CLI `--json`, audit reports, debugging) is weaker. Less reach for third-party tools that want to verify FFS atoms without an SDK.
- **Why rejected**: Human inspectability and tooling reach matter more at MVP scale than wire-size optimization. Phase 2+ may add a CBOR transport layer for federation while keeping JSON as the canonical form.

### Alternative 2: Protobuf with deterministic encoding

- **Description**: Define an `Atom` proto; sign the serialized bytes.
- **Pros**: Smallest, fastest. Strong typing.
- **Cons**: Canonical encoding is not guaranteed across protobuf implementations; the spec leaves field ordering and packed-vs-unpacked optional. Multi-language interop becomes a footgun for a substrate whose value depends on bit-exact reproduction. Schema evolution requires discipline; adding fields can break canonicalization.
- **Why rejected**: Canonical-encoding correctness is harder than it looks for protobuf. The signature/hash binding requires bit-exact agreement; JSON+JCS gives that with mature, well-tested implementations.

### Alternative 3: Plain JSON with field-order preservation by convention

- **Description**: JSON with documented field order and serialization rules but not formal canonicalization.
- **Pros**: Simplest. Most widely tooled.
- **Cons**: Different language libraries reorder keys, drop trailing zeros, escape strings differently. Verification fails on round-trips through any unaware tool. Convention without specification is not a contract.
- **Why rejected**: Bit-exact agreement is a hard requirement; convention does not produce it.

## Consequences

### Positive

- Human-readable atoms. `cat` and `jq` work. The auditor's daily report is legible.
- Mature JCS implementations exist in Rust (`serde_jcs`), TypeScript (`canonicalize`), and Python (`jcs`).
- Verification is portable: any language with a JSON parser, an Ed25519 verifier, and a JCS implementation can verify FFS atoms. Reach is wide.
- The `ffs://my-graph/atom/<hash>` contract is well-defined and durable.

### Negative

- Larger payloads than CBOR or protobuf (typically 1.3-2x). Federation pulls and SQLite storage carry the cost.
- JCS canonicalization has subtle rules (Unicode normalization, number representation); bugs exist in the wild but the spec is mature.

### Risks

- A JCS implementation diverges from the RFC and produces incompatible canonical bytes. **Mitigation**: pin a single Rust implementation as the reference; cross-language test vectors from RFC 8785 are checked in CI; the auditor verifies signatures on every atom it encounters.
- The envelope shape needs evolution. **Mitigation**: the `v` field is the schema version; the daemon refuses unknown versions; envelope evolution is forward-only with documented migration semantics.

## Implementation Notes

- Reference JCS implementation: `serde_jcs` crate in Rust.
- Numbers in atoms (timestamps, valid windows) use ISO 8601 strings, not Unix epochs, to avoid integer-vs-float ambiguity in JCS.
- Multihash codec ID for BLAKE3 is `0x1e`; the on-disk multihash encoding is `varint(0x1e) || varint(32) || blake3_digest`.
- The CLI's `--json` output emits the canonical envelope. The Obsidian plugin stores envelopes received from the daemon as opaque strings.
