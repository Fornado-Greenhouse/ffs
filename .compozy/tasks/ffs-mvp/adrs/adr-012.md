# ADR-012: Bilateral federation in MVP, multi-peer aggregation in Phase 2

## Status

Accepted

## Date

2026-05-05

## Context

Personal federation in FFS works through bilateral capability authoring: I author capabilities granting peer X tier-scoped views of my graph; X authors capabilities granting me views of theirs. Atoms cross filtered by capability at the source.

The MVP federation feature set has five components: bilateral capability authoring, peer mounting, tier-based selective sharing, intersection computation, revocation. A sixth feature was considered: **multi-peer aggregation** — a path that shows intersection or union across many peers in one view, not just bilateral.

Multi-peer aggregation is the most directly community-oriented federation feature. The 5-20 person small-community success criterion suggests it might belong in MVP. The Substrate-First MVP commitment (ADR-003) suggests deferring it as polish over a working primitive.

## Decision

MVP ships bilateral federation between pairs of peers. Each pair authors capabilities; intersection is computed pairwise; views are mounted pairwise.

Multi-peer aggregation — one path showing intersection or union across many peers — is Phase 2. It is UX leverage on top of the bilateral primitive; the primitive must be validated first.

## Alternatives Considered

### Alternative 1: Multi-peer aggregation in MVP

- **Description:** Ship the full community-oriented federation experience in MVP. A path like `~/.ffs/contacts/intersection/community/` shows intersection across all current peers; `~/.ffs/contacts/from/community/` shows the union of contacts the user has access to across peers.
- **Pros:** Most directly serves the success criterion. The community sees emergent value (the multi-peer intersection is where introductions get surfaced) earlier.
- **Cons:** Real complexity beyond the bilateral primitive — query optimization across many peers, intersection semantics with overlapping access tiers, performance at small-community scale. Building leverage before validating the primitive is premature.
- **Why rejected:** Substrate-First MVP discipline. Bilateral validates the architecture; multi-peer is polish.

### Alternative 2: Only bilateral, no aggregation ever

- **Description:** Ship bilateral federation. Multi-peer aggregation is permanently out of scope.
- **Pros:** Simpler architectural commitment.
- **Cons:** The community success criterion's "emergent introduction" indicator presumes some way for the user to find people across multiple peers' graphs at once. Without aggregation, this requires N pairwise navigation steps. That's tractable for 5-20 peers but loses the leverage that makes the substrate distinctively valuable.
- **Why rejected:** Multi-peer aggregation is real value that should ship eventually. Phase 2 is the right place; MVP just isn't.

## Consequences

### Positive

- MVP federation scope is sized to validate the primitive without over-investing in features that depend on the primitive working.
- Bilateral federation works for any number of peers — the user can have N bilateral relationships in MVP, just navigated pairwise.
- Phase 2 multi-peer aggregation builds on the proven bilateral foundation rather than being co-built with it.

### Negative

- Community-scale intersection ("who do all of us collectively know") requires N pairwise navigation steps in MVP, not one query. For a 5-20 person community this is tractable but tedious.
- The community-oriented emergent-introduction success indicator is harder to surface in MVP. Users have to discover intersections through pairwise checking.

### Risks

- Multi-peer aggregation turns out to be the actual community-defining feature, and MVP without it doesn't get adopted. **Mitigation:** Bilateral is the architectural primitive; if MVP retention metrics indicate aggregation is needed urgently, Phase 2 can prioritize it. If MVP retention is fine, aggregation is correctly deferred.

## Implementation Notes

Bilateral federation features in MVP:

- **Capability authoring.** I author a capability atom granting peer X specific tier-scoped views of my graph. The capability is bitemporal (valid windows), revocable (revocation is supersession), and signed.
- **Peer mounting.** X's contacts appear in `~/.ffs/contacts/from/<x>/` once the bilateral capabilities are exchanged.
- **Tier-based selective sharing.** The capability conditions support tier scopes — existence only, work email only, full notes, etc.
- **Intersection computation.** A path `~/.ffs/contacts/intersection/with/<x>/` lists contacts both I and X have at the access tier each granted.
- **Revocation.** Superseding a capability atom removes the peer's access; their view disappears; the audit trail records it.

Phase 2 adds multi-peer aggregation: paths like `~/.ffs/contacts/intersection/community/` and union views across all current peers, with appropriate query optimization.
