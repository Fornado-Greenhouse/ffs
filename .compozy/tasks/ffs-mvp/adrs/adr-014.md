# ADR-014: Minimum-viable fast-path for trivial projection edits in MVP

## Status

Accepted

## Date

2026-05-05

## Context

ADR-005 commits to editor-agnostic working-set materialization: projection files are real files on disk that any editor (Obsidian, Notepad, Vim, VS Code) can open. The original draft of ADR-005 specified that all edits to projection files route to the ingest folder as correction notebook entries, deferring the fast-path that authors atoms immediately to Phase 2.

This created an awkward inconsistency. The user opens `Sara_Chen.md` in Notepad, fixes a typo, saves — and the projection file gets reverted (or the underlying atom doesn't change, depending on how slow-path absorption is implemented). Their edit doesn't appear reflected. They'll think it didn't work; they'll re-edit; they'll either give up or open Obsidian to figure out what's happening.

The editor-agnostic commitment is half-realized if it works for reads but not for writes.

The fast-path resolves the inconsistency: trivial edits become atoms within milliseconds; the projection re-renders to reflect them; the user sees their fix immediately. But the full Phase 2 vision of the fast-path (semantic diff against rendered projections, reverse-map annotations on every predicate spec, optimistic UI with rollback, pending-edit atoms for ambiguous cases, system tray surface for cross-tool notifications) is substantial scope.

A minimum-viable fast-path is achievable in MVP because the scope of what counts as "trivial" can be tightly bounded.

## Decision

The MVP ships a minimum-viable fast-path for direct projection editing. The fast-path handles three categories of edits, for the three MVP predicate types only:

- **Single-line text-field edits.** Editing a value rendered from a single atom field (e.g., changing "Sarah" to "Sara" in a contact's name) is detected and authored as a supersession atom.
- **Frontmatter value edits.** Changing a frontmatter value (e.g., `tier: introducible` to `tier: discreet`) is detected and authored as a supersession on the corresponding classification or property atom.
- **Additive edits to designated sections.** Adding a new bullet to a section marked as additive in the predicate's rendering convention (e.g., a new note line in the Notes section of a contact) is authored as a new atom.

Edits the fast-path does not handle — multi-paragraph restructuring, substantial deletions, edits that span multiple atoms ambiguously, edits to federated projections (peer contacts), edits to projections rendered from predicates without reverse-map annotations — route to the ingest folder as correction notebook entries (the existing slow-path). The user reviews these in their daily health summary; the scribe proposes atoms; on acceptance they become supersessions.

Reverse-map annotations on the three MVP predicate specs (contact-related person predicates, generic person predicates, note predicates) make fast-path inference reliable. Maybe 15-20 reverse-map rules across all three.

## Alternatives Considered

### Alternative 1: Defer the fast-path to Phase 2 (the original draft of ADR-005)

- **Description:** All projection-file edits route to ingest as corrections in MVP. Fast-path ships in Phase 2.
- **Pros:** Smaller MVP scope. Fast-path can be designed comprehensively in Phase 2 with full reverse-map coverage.
- **Cons:** Editor-agnostic commitment is half-realized in MVP. Notepad users save edits that don't appear reflected; they think their edits failed. The Notepad-and-Explorer scenario (which ADR-005 commits to) is operationally broken without the fast-path.
- **Why rejected:** Inconsistency with ADR-005's editor-agnostic commitment. Either both ADRs (-005 and the fast-path deferral) ship together, or one of them needs to give. Shipping ADR-005's full file-system-shaped commitment without the fast-path produces a worse user experience than shipping neither.

### Alternative 2: Ship the full Phase 2 fast-path in MVP

- **Description:** Comprehensive fast-path for all edits, including ambiguous cases via the staged-edit pattern, with system tray surface for non-Obsidian notifications. Reverse-map annotations on all current and future predicate specs.
- **Pros:** Most complete editor-agnostic commitment.
- **Cons:** Substantial scope. The staged-edit pattern, the system tray surface, the comprehensive reverse-map work for predicates beyond MVP — all of this is Phase 2 polish, not MVP necessity. Shipping it all in MVP delays the small-community validation timeline without proportional benefit.
- **Why rejected:** YAGNI for MVP. The minimum-viable scope handles the trivial cases that account for most edits; ambiguous cases route through the existing slow-path which already works correctly.

### Alternative 3: Fast-path only for specific high-value cases (e.g., name and email edits only)

- **Description:** Even narrower scope — ship fast-path for the two or three highest-frequency edit cases on contacts, defer everything else.
- **Pros:** Smallest possible scope.
- **Cons:** The reverse-map annotation work doesn't shrink meaningfully (defining 5 rules vs. 15 rules is a small implementation difference; the daemon machinery is the same). The user-facing experience becomes inconsistent — some edits feel instant, others feel queued, and there's no obvious pattern from the user's perspective to predict which is which.
- **Why rejected:** Inconsistent UX is worse than uniform UX even if the uniform version is slightly more work. Three categories (single-line text fields, frontmatter values, additive sections) are coherent enough for users to develop accurate expectations.

## Consequences

### Positive

- The editor-agnostic commitment (ADR-005) is fully realized in MVP for both reads and writes.
- Notepad users see their typo fixes reflected immediately. The Notepad-and-Explorer scenario works as the architecture's commitments imply it should.
- Fast-path coverage is bounded and tractable: three predicate types, three edit categories, ~15-20 reverse-map rules, daemon machinery that classifies the diff and either authors an atom or falls back to slow-path.
- Phase 2 fast-path expansion (staged-edit pattern, system tray surface, broader predicate coverage) builds on a working MVP foundation rather than being co-built with the substrate.

### Negative

- MVP scope grows. Reverse-map annotations on predicate specs, daemon-side fast-path classification logic, optimistic UI rendering, and rollback handling are real implementation work.
- The fast-path's behavior depends on predicate-spec correctness. A bug in a reverse-map rule could cause atoms to be authored for the wrong field; this needs careful testing.

### Risks

- Reverse-map rules are subtle and getting them wrong is hard to detect. **Mitigation:** Test cases for each MVP predicate type cover the canonical edit patterns; reverse-map rules are inspectable and superseded the same way other predicate-spec components are.
- Fast-path absorption may create an unexpected number of supersession atoms (every typo fix is a supersession). **Mitigation:** This is correct behavior — supersession captures history. Compaction in Phase 2+ aggregates rapid same-field supersessions if they become operationally noisy.
- Optimistic UI rollback on capability failure may surprise users (the edit appears, then disappears). **Mitigation:** Capability failures should be rare in MVP since users author within their own substrates; failures route to a clear error message rather than silent rollback.

## Implementation Notes

Reverse-map annotations live in the predicate spec format alongside rendering rules. For each rendered output element (frontmatter field, body section, list item), the spec specifies which atom field the element maps back to. The daemon's fast-path uses these annotations to classify diffs.

The daemon detects file changes via filesystem watchers (ReadDirectoryChangesW on Windows, kqueue on macOS, inotify on Linux). On detection, the daemon parses the changed file against its last-rendered version, classifies the diff against the reverse-map rules for the relevant predicate, and either authors a supersession atom (fast-path) or routes to ingest as a correction (slow-path).

Optimistic UI rendering happens in the Obsidian plugin: the plugin updates the displayed view immediately on save, with rollback if the substrate write fails. For Notepad and other external editors, the daemon writes the canonical version back to disk after successful substrate write; editors that detect external file changes (Notepad on Windows 11, VS Code, Sublime) reload to the canonical version.
